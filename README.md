# 低轨卫星网络分布式动态路由研究思路

老师您好，这里是我目前对研究方向的一个初步梳理。我想把题目先收敛到：

**基于 PS-MAPPO 的队列感知与链路寿命约束低轨卫星网络分布式动态路由研究**

我目前的理解是，这个方向不能只写成“用强化学习做卫星路由”，这样会太宽。所以我想把问题聚焦到一个更具体的点：

**在低轨卫星高速运动、拓扑频繁变化的情况下，每颗卫星只依赖自身和一跳邻居信息，如何选择一个低拥塞、低时延、并且不容易失效的下一跳。**

---

## 1. 我想解决的具体问题

低轨卫星网络和普通地面网络不太一样，主要有几个特点：

- 卫星高速运动，星间链路会频繁变化；
- 网络规模大，如果一直依赖地面中心收集全网状态，控制开销会比较高；
- 业务流量可能不均匀，部分卫星或链路会出现拥塞；
- 某些链路虽然当前可用，但可能很快断开。

所以我目前想解决的问题可以概括为：

> 在局部信息受限的条件下，如何让每颗卫星根据邻居队列、链路负载和链路剩余可用时间，分布式地选择下一跳，从而降低端到端时延、丢包率和路由震荡。

这里面我重点关注三个痛点：

1. **局部信息下的快速路由决策**  
   每颗卫星不掌握全网状态，只能依赖自身和邻居信息做下一跳选择。

2. **队列积压和链路拥塞**  
   最短路径不一定是实际最快路径。如果下一跳队列很长，或者链路已经很忙，即使传播时延低，也可能造成更大的端到端时延。

3. **链路即将断开的问题**  
   LEO 星间链路具有明显的时间动态性。当前可用的链路可能很快失效，如果路由选择了这种链路，可能导致重路由、丢包和额外时延。

---

## 2. 整体建模思路

我打算先从网络本身开始建模，然后再引出多智能体强化学习。

### 2.1 时变图建模

因为 LEO 网络拓扑随时间变化，所以可以先把网络建成一个时变图：

```text
G(t) = (V, E(t), X(t))
```

其中：

- `V` 表示卫星节点集合；
- `E(t)` 表示 t 时刻可用的星间链路；
- `X(t)` 表示节点和链路状态，比如队列、带宽、负载、可靠性、链路剩余可用时间等。

这样建模的原因是：LEO 网络最核心的特点就是链路会随着时间变化，不能简单当作静态图处理。

### 2.2 智能体建模

每颗卫星看作一个智能体。卫星在转发数据包时，只需要决定当前数据包的下一跳，而不是一次性计算完整路径。

这样做的好处是：

- 更符合分布式执行；
- 可以根据实时队列和链路状态调整；
- 不需要频繁维护全局端到端路径。

### 2.3 局部观测设计

每颗卫星能够观测的信息主要包括：

- 自身队列长度；
- 邻居卫星队列长度；
- 相邻链路传播时延；
- 链路剩余带宽；
- 链路负载率；
- 链路可靠性或中断概率；
- 链路剩余可用时间；
- 当前数据包的目的节点或目的区域。

我加入这些状态的原因是：

- 队列长度用于判断下一跳是否拥塞；
- 链路负载用于判断链路是否繁忙；
- 传播时延用于判断基础传输距离；
- 剩余带宽用于判断链路传输能力；
- 可靠性用于判断传输失败风险；
- 链路剩余可用时间用于判断链路是否快要断开。

### 2.4 动作设计

动作就是选择下一跳：

```text
a_i(t) = 选择某个邻居卫星作为下一跳
```

如果当前卫星有多个邻居，就从这些邻居里面选择一个。

### 2.5 动作过滤

在真正选择下一跳之前，我想先做一层动作过滤。也就是先把明显不适合的链路排除掉，比如：

- 链路当前不可用；
- 剩余带宽不足；
- 下一跳队列过长；
- 链路可靠性过低；
- 链路剩余可用时间太短；
- 可能形成路由环路。

这样做的原因是，强化学习不应该完全从零开始学习这些明显的硬约束。先过滤掉不可行动作，可以减少无效探索，也能提高路由稳定性。

## 3. 方法选择：PS-MAPPO

我目前打算采用 **PS-MAPPO** 作为具体方法。这里 PS-MAPPO 指：

**Parameter Sharing Multi-Agent Proximal Policy Optimization**

也就是：

```text
参数共享 Actor + 集中式 Critic + 分布式执行 + 动作掩码
```

选择这个方法的原因是：

1. **这是一个多智能体问题**  
   每颗卫星的转发决策都会影响其他卫星的队列和链路负载，所以不能只看单个节点的局部最优。

2. **适合 CTDE 思路**  
   训练阶段可以在地面仿真平台获取全局状态，所以可以使用集中式 Critic；但真正执行时，每颗卫星只能用本地观测，所以只部署 Actor。

3. **卫星节点比较同构**  
   大多数卫星的转发功能类似，因此可以让所有卫星共享同一个 Actor 网络，减少模型参数量和星上存储开销。

4. **动作是离散的**  
   路由动作就是从邻居中选择一个下一跳，比较适合 MAPPO 这种离散动作策略方法。

5. **方便加入动作掩码**  
   LEO 网络中邻居集合会变化，部分链路也会不可用，所以动作掩码比较重要。

训练阶段：

- Critic 可以看到全局网络状态；
- 用全局奖励引导多颗卫星协同；
- 目标是降低全网时延、队列积压、丢包率和路由切换。

执行阶段：

- 每颗卫星只使用共享 Actor；
- 输入是局部观测和邻居状态；
- 输出是下一跳选择；
- 不依赖地面中心实时控制。

## 4. 奖励函数设计思路

奖励函数我想分成两层考虑。

### 4.1 全局协同奖励

训练时可以使用全局指标，例如：

- 平均端到端时延越低越好；
- 全网队列积压越小越好；
- 链路负载越均衡越好；
- 丢包率越低越好；
- 路由切换次数越少越好；
- 吞吐量越高越好。

这样做是为了避免每颗卫星只追求局部最优，导致全局拥塞。

### 4.2 局部即时奖励

局部奖励主要用于引导单跳选择，例如：

- 选择低传播时延链路，奖励更高；
- 选择队列较短的邻居，奖励更高；
- 选择高负载链路，给予惩罚；
- 选择可靠性低或即将断开的链路，给予惩罚；
- 形成环路，给予惩罚；
- 数据包成功到达目的地，给予正奖励。

这样设计的目标是让智能体学会：不仅要走得近，还要避开拥塞和快断链路。

## 5. 预期创新点

目前我认为可以提炼成三个创新点：

1. **队列感知的分布式路由决策**  
   在下一跳选择中加入邻居队列长度和链路负载，避免流量集中到局部最短路径。

2. **链路寿命约束的动作过滤机制**  
   在策略网络选择动作前，过滤即将断开或可靠性较低的链路，减少路径中断和重路由。

3. **基于 PS-MAPPO 的 CTDE 协同路由框架**  
   训练阶段利用全局状态学习协同策略，执行阶段每颗卫星只根据局部信息独立决策。

## 6. 实验验证思路

实验部分我打算从两个角度验证：整体性能和模块贡献。

### 6.1 对比算法

可以考虑和以下方法对比：

- Dijkstra 最短路径；
- 虚拟拓扑路由；
- Q-routing 或 QRLSN；
- 普通 MAPPO；
- 本文完整方法。

### 6.2 实验场景

计划设置几类典型场景：

1. **低负载场景**  
   验证本文方法在轻负载下不会明显劣于最短路径。

2. **高负载热点场景**  
   验证队列感知和负载感知是否能缓解拥塞。

3. **链路频繁断开场景**  
   验证链路剩余可用时间约束是否能降低丢包和重路由。

4. **节点或链路故障场景**  
   验证算法在异常情况下的鲁棒性。

### 6.3 评价指标

主要指标包括：

- 平均端到端时延；
- 吞吐量；
- 投递成功率；
- 丢包率；
- 平均队列长度；
- 链路负载均衡度；
- 路由切换次数；
- 控制消息开销。

### 6.4 消融实验

为了证明每个模块不是简单堆上去的，可以做消融实验：

- 去掉队列感知；
- 去掉链路负载项；
- 去掉链路剩余可用时间；
- 去掉动作掩码；
- 去掉全局协同奖励。

通过这些实验观察端到端时延、丢包率、队列长度和路由稳定性是否变差，从而说明各模块的作用。

## 8. 目前总结

目前我的整体思路是：

> 先把 LEO 卫星网络建模为时变图，再把每颗卫星建模为一个智能体。每颗卫星只利用自身状态、一跳邻居状态和相邻链路状态进行下一跳选择。为了避免拥塞和链路失效，在状态和奖励中加入队列长度、链路负载、链路可靠性和链路剩余可用时间，并通过动作掩码过滤明显不可用或高风险链路。方法上采用 PS-MAPPO，实现训练阶段全局协同、执行阶段分布式自治。实验上通过对比实验和消融实验验证该方法在时延、丢包率、负载均衡和路由稳定性方面的效果。

这还是一个初步方案，后续需要进一步细化的地方包括：

- 状态空间具体如何归一化和组织；
- 链路剩余可用时间如何计算；
- 奖励函数各项权重如何设置；
- 仿真星座和业务流量如何生成；
- 与已有 LEO 路由算法的公平对比方式。
