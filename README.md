# 低轨卫星网络分布式动态路由研究思路
题目先收敛到：
基于 RTMDP 建模的队列感知与链路寿命约束低轨卫星网络分布式动态路由研究
## 摘要
本文研究的是低轨卫星网络中的分布式动态路由问题。LEO 卫星移动速度快、星间链路频繁变化，且大规模星座下全网链路状态同步开销较高，因此不适合完全依赖地面中心实时计算全局最短路。本文把每颗卫星看作一个智能体，只利用自身状态、一跳邻居状态和相邻链路状态进行下一跳选择。
具体思路是：参考 RTMDP 的实时路由建模方式，把一次路由决策表示成“当前状态、可选动作、状态转移和即时奖励”的过程。在状态中显式加入邻居队列长度、链路负载、链路可靠性和链路剩余可用时间，避免只按传播时延选路导致拥塞或选到快断链路。求解方法上采用 MAPPO/CTDE 思路：训练阶段使用集中式 Critic 和全局奖励学习协同策略，执行阶段每颗卫星只根据局部观测独立选择下一跳。
预期验证重点包括三点：第一，队列和负载感知能否降低最大队列长度和 95% 分位时延；第二，链路寿命约束能否减少丢包、重路由和路由切换；第三，MAPPO/CTDE 的协同训练能否在不显著增加控制开销的情况下，提高投递成功率、吞吐量和负载均衡效果。

## 1. 我想解决的具体问题
低轨卫星网络和普通地面网络不太一样。这个场景里最麻烦的不是“能不能算一条最短路”，而是这个最短路在高动态和高负载情况下还靠不靠谱。
场景特点：
- 规模很大：【文献/公开资料】以 Starlink 为例，Space.com 在 2026 年 6 月上旬的报道中提到，Starlink 活跃卫星数量已经超过 10,500 颗；2026 年 6 月 8 日的发射报道中又提到已超过 10,580 颗。这里主要说明“万颗级星座已经是现实规模”，不把 10,580 当成固定常数。
参考：https://www.space.com/spacex-starlink-satellites.html
- 移动很快：【文献/公开资料】低轨卫星维持轨道的速度量级通常在 7.8 km/s 左右，也就是约 28,000 km/h。这个数值会随轨道高度变化，但已经足够说明：星间链路和星地可见关系不能按静态网络理解。
参考：https://en.wikipedia.org/wiki/Low_Earth_orbit
- 集中式路由开销确实是问题：【文献结果】Page 等人在 Distributed Probabilistic Congestion Control in LEO Satellite Networks 中指出，在密集 LEO 星座中使用集中式最小时延路由会带来显著的信令和计算开销，因此转向基于邻居最新流量信息的分布式拥塞控制。具体开销大小还取决于星座规模、更新周期和状态字段设计。
参考：https://arxiv.org/abs/2209.08565
- 队列和实时流量变化不能忽略：【文献结果】2026 年的 Queue-Aware and Resilient Routing in LEO Satellite Networks Using Multi-Agent Reinforcement Learning 里提到，传统路由通常假设静态链路指标，不能很好处理 queue backlogs 和实时系统变化；同一篇文章还报告，在它的实验设置中，5 s 重计算间隔下所提方法的开销约为 Dijkstra 的 50%。这个结果只能说明“学习型局部决策有降低开销的可能”，不能直接推广成所有 LEO 场景都一定减半。
参考：https://arxiv.org/abs/2605.04448
- 热点区域会放大拥塞问题：【文献结果】Cao 等人在 Computing-Aware Routing for LEO Satellite Networks 中提到，热点区域会让可见 LEO 卫星的有限计算/传输资源更加紧张；他们的 computing-aware routing 在该文仿真设置下，相比把原始数据直接下传到地面处理，整体时延最高下降 78.31%。虽然这篇重点是计算感知路由，不是单纯的数据包路由，但它能支撑一个判断：热点区域和资源受限会明显改变路径选择效果。
参考：https://arxiv.org/abs/2211.08820
本研究的具体问题是：
在局部信息受限的条件下，如何让每颗卫星根据邻居队列、链路负载和链路剩余可用时间，分布式地选择下一跳，从而降低端到端时延、丢包率和路由震荡。
重点痛点：
1. 局部信息下的快速路由决策
万颗级星座里，如果每次拓扑或流量变化都依赖全局状态收集，再集中计算最短路径，信令和计算开销会很高。已有文献也指出密集 LEO 星座中集中式最小时延路由存在显著开销问题。因此本文只让每颗卫星依赖自身和一跳邻居信息做下一跳选择。
2. 队列积压和链路拥塞
最短路径不一定是实际最快路径。如果下一跳队列很长，或者链路已经很忙，即使传播时延低，也可能造成更大的端到端时延。Queue-aware LEO routing 的文献也说明，传统静态链路指标没有充分考虑 queue backlogs 和实时流量变化，这会影响路由效果。
3. 链路即将断开的问题
LEO 星间链路具有明显的时间动态性。当前可用的链路可能很快失效，如果路由选择了这种链路，可能导致重路由、丢包和额外时延。因此链路剩余可用时间需要进入动作过滤和奖励设计。
核心不是“最短路怎么求”，而是：
在局部信息、实时拥塞和链路高动态同时存在的情况下，怎么让卫星更稳地选下一跳。

### 2.1 时变图建模
LEO 网络拓扑随时间变化，可建模为时变图：
```text
G(t) = (V, E(t), X(t))
```
其中：
- V 表示卫星节点集合；
- E(t) 表示 t 时刻可用的星间链路；
- X(t) 表示节点和链路状态，比如队列、带宽、负载、可靠性、链路剩余可用时间等。
这样处理是因为 LEO 链路随时间变化，不能简单当作静态图。
“控制开销大”可以用一个数量级例子说明。【估算】假设万颗级星座里有 N = 10,000 颗卫星，每颗卫星平均维护 4 条星间链路，那么无向链路数大约是：
```text
|E| ≈ N × 4 / 2 = 20,000
```
如果每条链路状态至少包含：
```text
链路两端 ID、传播时延、剩余带宽、负载率、可靠性、链路剩余时间、时间戳
```
【估算】即使非常粗略地按每条链路 32 B 计算，一次全网链路状态快照也大约是：
```text
20,000 × 32 B ≈ 640 KB
```
这个 640 KB 只是原始链路状态，不包括协议头、转发冗余、可靠广播开销和多跳传播。如果要求每颗卫星都拿到全网状态，那么一次全网同步的总分发量可以达到：
```text
640 KB × 10,000 ≈ 6.4 GB
```
如果每 5 s 更新一次，纯状态分发的数量级就是：
```text
6.4 GB / 5 s ≈ 1.28 GB/s
```
【估算】上面这个计算不是某篇论文的实验值，只是说明数量级：在万颗级星座里，全网链路状态同步很容易变成较重的控制负担。【文献结果】2605.04448 那篇 queue-aware MARL 文章也报告，在该文设置的 5 s 重计算间隔下，它的方法开销约为 Dijkstra 的 50%。全局最短路类方法的重计算和状态维护可能成为瓶颈，具体程度需要通过本文仿真实验确认。
### 2.2 智能体建模
每颗卫星看作一个智能体。卫星在转发数据包时，只需要决定当前数据包的下一跳，而不是一次性计算完整路径。
优点：
- 更符合分布式执行；
- 可以根据实时队列和链路状态调整；
- 不需要频繁维护全局端到端路径。
逐跳路由不保证每一步都得到全局最短路径，但更适合高动态场景。提前计算完整路径时，路径中间某条链路断开就需要重算；逐跳路由虽然可能局部绕路，但每一步都能根据当前状态重新判断。
需要验证的不是“每条路径是不是最短”，而是：
```text
逐跳分布式决策能不能在总体上降低丢包、队列积压和链路失效带来的重路由。
```
如果平均跳数略有增加，但 95% 分位时延、丢包率和重路由次数下降，则说明这个取舍是合理的。
### 2.3 局部观测设计
每颗卫星能够观测的信息主要包括：
- 自身队列长度；
- 邻居卫星队列长度；
- 相邻链路传播时延；
- 链路剩余带宽；
- 链路负载率；
- 链路可靠性或中断概率；
- 链路剩余可用时间；
- 当前数据包的目的节点或目的区域。
这些状态分别对应：
- 队列长度用于判断下一跳是否拥塞；
- 链路负载用于判断链路是否繁忙；
- 传播时延用于判断基础传输距离；
- 剩余带宽用于判断链路传输能力；
- 可靠性用于判断传输失败风险；
- 链路剩余可用时间用于判断链路是否快要断开。
局部观测可以拆成三组：
```text
自身状态：q_i, buffer_i, service_rate_i
邻居状态：q_j, load_j, available_j
链路状态：delay_ij, bandwidth_ij, rho_ij, reliability_ij, T_rem_ij
```
其中最关键的是 q_j 和 T_rem_ij：
- q_j 解决的是“下一跳现在堵不堵”；
- T_rem_ij 解决的是“这条链路会不会很快断”。
2605.04448 那篇文章里把单跳时延写成传播时延、传输时延和排队时延的组合，说明队列项会直接进入时延。如果只看传播距离，实际可能选到一条“距离短但队列很长”的路径。

### 2.4 动作设计
动作就是选择下一跳：
```text
a_i(t) = 选择某个邻居卫星作为下一跳
```
如果当前卫星有多个邻居，就从这些邻居里面选择一个。
### 2.5 动作过滤
真正选择下一跳之前，先做一层动作过滤，把明显不适合的链路排除掉：
- 链路当前不可用；
- 剩余带宽不足；
- 下一跳队列过长；
- 链路可靠性过低；
- 链路剩余可用时间太短；
- 可能形成路由环路。
强化学习不需要从零学习这些硬约束。先过滤不可行动作，可以减少无效探索，也能提高路由稳定性。
动作过滤可以写成下面这种规则：
候选下一跳需要满足六个条件：
1. 这条链路现在真的能用；
2. 这条链路剩余带宽不能太低；
3. 下一跳卫星不能已经排了很长队；
4. 这条链路不能太不可靠；
5. 这条链路不能马上就断；
6. 这个下一跳不能让数据包绕回已经走过的节点。
对应成符号就是：
```text
候选下一跳 j 必须同时满足：
1. link_ij 当前可用
2. bandwidth_ij > B_min
3. q_j < Q_max
4. reliability_ij > R_min
5. T_rem_ij > T_safe
6. j 不在当前数据包已访问节点集合中
```
其中 T_safe 不应随意设为常数，至少要大于一次转发可能需要的时间：
```text
T_safe > T_queue + T_transmission + T_propagation + T_margin
```
如果链路剩余可用时间不足以支撑当前数据包安全通过，即使它当前可用，也不应该选。
这一点区别于直接让策略网络在所有邻居里选的做法：先用网络约束做 action mask，再让策略在可行动作里选择。

## 3. 方法选择：实时 Dec-POMDP 建模 + MAPPO/CTDE 求解
RTMDP 的核心招式是用【在途动作】进行增广，直观讲：你这一步算出来的动作 a_i^t，因为有延迟 δ，其实是作在下一个状态 s^{t+1} 上的。RTMDP 的做法是把「上一步算出、正在途中的动作」也当成状态的一部分，这样「算动作时状态在变」就被显式建出来了。
更改后把方法分成三层：
```text
第一层（问题是什么）：Dec-POMDP，刻画"多卫星协作 + 局部可观测"
第二层（实时性）：RTMDP实时增强，刻画决策延迟、观测时效与误差
第三层（怎么求解）：CTDE范式 + MAPPO算法
```
第一层的Dec-POMDP：有很多颗卫星（多智能体），每颗只看得到自己 + 一跳邻居（部分可观测），大家为同一个全局目标协作（共享奖励），解决MDP/RTMDP “一个智能体 + 看得到全局”这个问题
第二层RTMDP：Dec-POMDP 默认“算动作不花时间、状态算的时候不变”，用RTMDP去修这个【实时】漏洞
第三层CTDE/MAPPO
### 3.1 实时 Dec-POMDP 建模
RTMDP 可以理解成实时路由场景下的 MDP 建模。它关注的不是一次性算完整路径，而是在每个转发时刻，根据当前网络状态立即选下一跳。
对应到 LEO 路由中，一次决策可以写成：
```text
状态 s_t：当前卫星、目的节点、邻居队列、链路负载、可靠性、剩余可用时间
动作 a_t：从当前卫星的邻居集合中选择一个下一跳
状态转移：数据包被转发到下一跳，队列和链路状态随时间更新
奖励 r_t：根据时延、队列、负载、丢包、链路稳定性给奖励或惩罚
```
这样建模的好处是，它正好对应 LEO 网络的实时性：链路状态和队列状态会不断变化，路由决策必须边转发边更新，而不是等全局路径周期性重算。

LEO 路由是一个多智能体、部分可观测问题：每颗卫星是一个智能体，执行时只能看到自身、一跳邻居和相邻链路，且所有卫星为同一全局目标协作。这超出了普通 MDP（单智能体、全可观测）的表达范围，因此在这里尝试用 Dec-POMDP 作为问题模型：
```text
Dec-POMDP = ⟨ I, S, {A_i}, P, R, {Ω_i}, O, γ ⟩
I    : 卫星智能体集合
S    : 全局状态（全网队列、链路、拓扑、包位置），仅训练期集中式 Critic 可见
A_i  : 卫星 i 的动作（在时变候选集 A_i(t) 中选择下一跳）
P    : 联合状态转移 P(s'|s,a)
R    : 协作型全局共享奖励（即 R_global）
Ω_i  : 卫星 i 的观测空间（自身 + 一跳邻居 + 相邻链路）
O    : 观测函数 O(o_i|s)，把全局真状态映射为局部、延迟、含噪的观测
γ    : 折扣因子
```
这里的关键是 S 与 O 的分离：真状态 S 是全局的，卫星执行时并不可见；卫星只能经由观测函数 O 得到局部观测 o_i，即 o_i ≠ S，这正是"部分可观测"的形式化表达。
在此基础上，针对星上实时决策的延迟与状态快变，本文采用 RTMDP的实时状态增强，把三类实时性显式写进模型：
```text
① 决策延迟：算一次动作需推理+传播延迟 δ，动作实际作用于 s^{t+1}。
   按 RTMDP，将"在途动作" a_i^{t-1} 纳入增广观测。
② 观测时效：邻居信息由 Hello 包携带，带信息年龄 age_j (Age of Information)，
   其分量反映 s_j^{t−Δ_j} 而非 s_j^t。
③ 观测误差：reliability、T_rem 为估计量，故 O 为随机观测函数 O(o_i|s)。
```
于是卫星 i 的增广观测为：
```text
õ_i^t = ( o_i^t ,  a_i^{t-1} ,  {age_j} )
```
由于部分可观测，最优策略依赖观测历史（信念状态）而非瞬时真状态。Dec-POMDP 与上述 RTMDP 实时增强合起来，构成本文的实时 Dec-POMDP 问题模型。
各数学对象与 LEO 路由的对应如下：
RTMDP 在本文中不再作为整体框架，而是作为实时观测增强机制；多智能体协作由 Dec-POMDP 承载，求解由后文 CTDE/MAPPO 完成。

### 3.2 为什么用 MAPPO/CTDE 求解
1. 这是一个多智能体问题
每颗卫星的转发决策都会影响其他卫星的队列和链路负载，所以不能只看单个节点的局部最优。
2. 适合 CTDE 思路
训练阶段可以在地面仿真平台获取全局状态，所以可以使用集中式 Critic；但真正执行时，每颗卫星只能用本地观测，所以只部署 Actor。
3. 卫星节点比较同构
大多数卫星的转发功能类似，因此 Actor 可以采用统一结构，后续再根据星上资源决定是否压缩模型。
4. 动作是离散的
路由动作就是从邻居中选择一个下一跳，比较适合 MAPPO 这种离散动作策略方法。
5. 方便加入动作掩码
LEO 网络中邻居集合会变化，部分链路也会不可用，所以动作掩码比较重要。
更具体地说，本文采用共享 Actor + 集中式 Critic：

```text
Actor:  π_theta(a_i | õ_i^t, mask_i(t))
Critic: V_phi(S^t) 或 Q_phi(S^t, a^t)
```

Actor 的输入只包含星上能拿到的增广局部观测和动作掩码，所以执行阶段可分布式部署；Critic 在训练时可以看到全局状态，用来学习多个卫星动作对全网指标的影响。
训练阶段：
- Critic 可以看到全局网络状态；
- 用全局奖励引导多颗卫星协同；
- 目标是降低全网时延、队列积压、丢包率和路由切换。
执行阶段：
- 每颗卫星只使用 Actor；
- 输入是局部观测和邻居状态；
- 输出是下一跳选择；
- 不依赖地面中心实时控制。

## 4. 奖励函数设计思路
奖励函数先把每个指标归一化到 [0, 1]，再区分正奖励和惩罚项，最后通过权重体现目标优先级。
奖励分成两层：
```text
训练阶段：全局协同奖励 R_global
单跳选择：局部即时奖励 r_local
```
为了保证多目标奖励中的权重能够表示目标优先级，而不是被指标单位和数值尺度影响

奖励函数不是直接把原始指标相加，而是先构造归一化后的代价项。设原始指标为 delay、queue、loss、reroute 和 overhead。由于这些指标的单位和数量级不同，先对每个指标进行无量纲化处理，再进行加权求和。

对于本身有明确范围的指标，例如丢包率、投递成功率、链路负载率和可靠性，直接使用 [0,1] 范围内的比例值。对于端到端时延、队列长度和控制开销这类可能出现长尾分布的指标，先在基线算法和不同流量负载下统计其分布，包括均值、方差、P50、P90、P95 和 P99，再选取参考值进行归一化。若指标存在明显极端值，则采用 percentile clipping 或 log scaling，避免少量异常样本主导奖励函数。
各指标怎么处理

### 4.1 全局协同奖励
全局奖励初步设计为：
```text
R_global =
  + w_T * S_throughput
  + w_S * S_delivery
  - w_L * C_delay
  - w_Q * C_queue
  - w_B * C_load_balance
  - w_P * C_loss
  - w_S * C_switch
  - w_O * C_overhead
```
S 表示 score，越大越好；C 表示 cost，越小越好。w 表示权重，例如 w_P 越大，表示越不能接受丢包；w_Q 越大，表示越希望避免队列积压。
直观上：
```text
奖励 = 好事情的加分 - 坏事情的扣分
```
好事情包括吞吐高、送达率高；坏事情包括时延高、队列长、负载不均衡、丢包、切换频繁、控制开销大。

举个例子，如果某个训练阶段里：
```text
S_throughput = 0.85

```
这表示网络已经有一定拥塞和时延压力。奖励会推动模型继续提高吞吐和投递成功率，同时压低队列、时延和丢包。
权重不直接定死，先按重要性排序：
```text
投递成功率/丢包  >  时延  >  队列/负载均衡
```
【实验前假设】
```text
w_S = 2.0   投递成功率
w_P = 2.0   丢包惩罚
w_L = 1.5   时延惩罚
w_Q = 1.0   队列惩罚
```
这样设置的原因是：如果包送不到，吞吐或时延再好也没有意义，所以投递成功率和丢包权重要更高。控制开销不能压过路由质量，否则模型可能为了少交换状态而选错路。
【待验证】权重敏感性实验：
- 增大 w_Q，看队列是否下降，但路径是否绕远；
- 增大 w_S，看路由是否更稳定，但拥塞恢复是否变慢；
- 增大 w_T，看吞吐是否提高，但丢包是否上升；
- 增大 w_O，看控制开销是否下降，但状态过期是否让性能变差。
【待验证】全局奖励的作用通过实验量化：
```text
完整方法 vs 去掉全局协同奖励
```
如果去掉全局奖励后，平均时延变化不大，但最大队列长度、链路负载方差和丢包率上升，则说明全局奖励主要作用在削峰和负载均衡，而不是只优化平均值。
### 4.2 局部即时奖励
局部即时奖励主要负责某一跳怎么选。【实验前假设】一个初步形式可以写成：
```text
r_local(i, j) =
  + eta_1 * I_deliver
  - eta_2 * delay_ij_norm
  - eta_3 * queue_j_norm
  - eta_4 * load_ij_norm
  - eta_5 * risk_ij
  - eta_6 * lifetime_penalty_ij
  - eta_7 * I_loop
  - eta_8 * I_switch
```

```text
if T_rem_ij < T_safe:
    直接 mask 掉
else:
    lifetime_penalty_ij = T_safe / T_rem_ij
```
比如：
```text
T_safe = 2 s

```
即：剩余时间低于安全阈值直接过滤，高于阈值但越接近阈值惩罚越大。
局部奖励的目标不是让每一跳都选最短，而是在下面四个因素之间折中：
```text
短路径 + 低队列 + 低负载 + 不快断
```
因此算法不一定选择传播时延最小的邻居；如果该邻居队列很长或链路快断，端到端结果反而可能更差。

## 5. 预期创新点
可以提炼成三个创新点：
1. 队列感知的分布式路由决策
改成：通信预算约束下的队列-负载感知局部观测设计
在下一跳选择中加入邻居队列长度和链路负载，避免流量集中到局部最短路径。
已有 queue-aware 方法证明了队列信息对拥塞感知有用，但本文进一步关注：在 Hello 包字段长度和交换频率受限时，哪些状态最值得交换，以及这些状态分别改善哪些全局指标
2. 链路寿命约束的动作过滤机制
在策略网络选择动作前，过滤即将断开或可靠性较低的链路，减少路径中断和重路由。
不假设卫星可以无限交换状态，而是把 Hello 包开销作为约束。核心问题变成：在固定控制开销预算下，应该交换哪些局部状态，才能获得最好的路由效果。
建立一个简单通信开销模型：
```text
overhead_ctrl = N * d_avg * L_hello / T_hello
```
其中：
- N 是卫星数量；
- d_avg 是平均邻居数；
- L_hello 是每个 Hello 包携带的状态字段长度；
- T_hello 是 Hello 周期。
如果状态字段更多，L_hello 增大；如果想降低开销，就要增大 T_hello，但信息会变旧。所以这里天然有一个 trade-off：
```text
状态字段越多 -> 单次信息更完整 -> 但 Hello 包更大
Hello 周期越短 -> 信息更新更及时 -> 但控制开销更高
Hello 周期越长 -> 控制开销更低 -> 但队列/负载信息更容易过期
```
用这几组实验：
最有说服力的指标不是单纯平均时延，而是：
```text
单位控制开销带来的性能收益 =
(基线指标 - 当前指标) / 控制开销
```
比如：
```text
delay_gain_per_KB = (D_baseline - D_method) / overhead_ctrl
```
3. 基于 RTMDP + MAPPO/CTDE 的协同路由框架
先用 RTMDP 明确状态、动作、转移和奖励，再用 MAPPO/CTDE 在训练阶段学习协同策略，执行阶段每颗卫星只根据局部信息独立决策。
每个创新点对应一组验证实验：
创新点
需要证明什么
对应实验
队列感知
加入队列后是否真的缓解拥塞
高负载热点场景，比较平均队列、最大队列、端到端时延
链路寿命约束
是否减少快断链路导致的失败
链路频繁断开场景，比较丢包率、重路由次数、路径中断次数
MAPPO/CTDE 协同路由
是否比单纯局部决策更均衡
去掉全局奖励/普通 MAPPO 消融，比较负载方差、吞吐量、投递成功率
队列感知有效时，不应只看到平均时延下降，还应看到最大队列长度下降；链路寿命约束有效时，不应只看到丢包率下降，还应看到链路断开导致的重路由次数减少。

## 8. 研究思维画布九步法梳理
这一节按“研究思维画布九步法”把整个课题重新串起来。它不是单纯重复前面的建模，而是说明：这个题目为什么值得做、具体卡在哪里、方案解决了哪一段问题、最后怎么证明。
### 8.1 用户场景
应用场景是高动态低轨卫星网络中的星间路由。研究对象不是单条链路，也不是单颗卫星，而是由大量 LEO 卫星组成的时变网络。
具体可以理解成下面这个转发过程：
```text
源地面站/用户
    ↓
接入卫星
    ↓
多跳星间链路转发
    ↓
目标区域附近卫星
    ↓
目的地面站/用户
```
在这个过程中，每颗中继卫星只负责一个很小的动作：给当前数据包选下一跳。但这个小动作会影响后面的队列、链路负载、丢包和重路由。
这个场景的特殊性在于：
特点
对路由的影响
卫星高速运动
星间链路和星地可见关系不断变化
星座规模大
全网状态收集和分发成本变高
流量不均匀
热点区域附近卫星容易排队
链路存在中断风险
当前可用链路可能很快不可用
星上资源有限
路由决策不能太复杂，也不能依赖大量控制消息
所以这个课题面向的不是“静态网络中求一条最短路”，而是“动态、拥塞、局部信息受限条件下的逐跳路由决策”。
### 8.2 研究需求
核心需求可以拆成一句话：
在低控制开销和低计算开销的前提下，让每颗卫星在线选出一个低时延、低拥塞、低失效风险的下一跳。
“低开销”不是只指控制消息少，而是包括两部分：
```text
1. 控制开销：卫星之间为了获得状态信息而交换的 Hello 包、链路状态更新、路由重计算触发消息等。
2. 计算开销：星上执行路由决策时需要的计算量，包括动作过滤复杂度、Actor 推理 FLOPs、单次决策时间、模型参数量和内存占用。
```
在低控制开销和低星上计算开销的约束下，让每颗卫星仅依赖本地观测和一跳邻居信息，在线选择一个低时延、低拥塞、低链路失效风险的下一跳。
两个开销分开定义
控制开销写成
```text
overhead_ctrl = N * d_avg * L_hello / T_hello
```
其中：
- N：卫星数量；
- d_avg：平均邻居数；
- L_hello：每个 Hello 包携带的状态字段长度；
- T_hello：Hello 周期。
这个公式说明：状态字段越多，L_hello 越大；Hello 越频繁，T_hello 越小，控制开销越高。
计算开销写成
执行阶段每颗卫星只做两步：
```text
1. 对候选邻居做动作过滤
2. 用 Actor 给剩余邻居打分并选择下一跳
```
动作过滤复杂度：
```text
O(k_i)
```
其中 k_i 是当前卫星候选邻居数。LEO 常规 ISL 场景里 k_i 通常很小，比如 4 条左右，所以过滤开销不随全网卫星数 N 增长。
Actor 推理复杂度可以写成：
```text
C_infer = k_i * C_actor
```
如果每个邻居使用共享 MLP 编码器，例如：
```text
z_ij: 7维
MLP: 7 -> 32 -> 16 -> 1
```
则每个邻居大约需要：
```text
MACs = 7*32 + 32*16 + 16*1 = 752
FLOPs ≈ 2 * 752 = 1504
```
如果 k_i = 4：
```text
总 FLOPs ≈ 4 * 1504 = 6016 FLOPs
```
这个数只是估算，用来说明：执行阶段复杂度主要跟邻居数和小型 Actor 网络有关，而不是跟全网规模 N 直接相关。
和 Dijkstra 的计算开销对比
这里的方法不一定比 Q-routing 更轻，但它比全局最短路类方法更适合大规模动态网络，即牺牲少量神经网络推理开销，换取更强的状态表达能力，并避免全网链路状态维护和全局路径重计算
实验指标要补这些
这里面有三个矛盾。
第一个矛盾是“全局最优目标”和“局部可观测信息”之间的矛盾。端到端时延、全网拥塞和负载均衡都是全局指标，但星上执行时每颗卫星只能看到自己和邻居。
第二个矛盾是“快速决策”和“状态新鲜度”之间的矛盾。更新越频繁，状态越新，但控制开销越大；更新越少，开销低，但卫星拿到的信息可能已经过期。
第三个矛盾是“短路径”和“稳定低拥塞路径”之间的矛盾。传播距离短不一定代表端到端时延低，因为下一跳可能队列很长，或者链路快断。
控制开销具体包括三类：
开销类型
来源
对本文的影响
状态收集开销
收集全网链路、队列、负载状态
星座越大越难实时收集
状态分发开销
将全网状态传播给所有节点
可能占用星间链路资源
重计算开销

【估算】按前面的粗略估算，万颗级星座一次全网链路状态快照就可能是数百 KB 级，如果分发给所有卫星，总量就是 GB 级；如果几秒一次更新，控制流量会很可观。因此本文更倾向于局部邻居交换，而不是全网实时状态同步。
对应到本文，就是：
```text
不用全网实时状态
    ↓
只依赖本地和邻居状态
    ↓
仍然尽量接近全局目标：低时延、少拥塞、少丢包、低开销
```
### 8.3 核心卖点
核心卖点不是“用了强化学习”，而是把 LEO 路由里几个容易被忽略的因素真正放进决策里。
1. 基于队列和负载状态进行下一跳决策
不只看传播时延，也看下一跳是否拥塞。如果邻居 j 的队列已经很长，即使 i -> j 链路传播时延低，也可能导致端到端时延升高。
2. 基于链路寿命约束进行动作过滤
在策略选择前先过滤掉即将断开、可靠性低或不可用的链路。这个设计主要针对 LEO 链路高动态问题。
3. 基于 MAPPO/CTDE 实现集中训练、分布式执行
训练时利用全局状态学习协同策略，执行时每颗卫星只用局部观测独立决策。这样既能利用全局奖励训练协作行为，又不要求星上执行时依赖地面中心实时控制。
三个卖点对应的“局部动作 -> 全局结果”关系如下。下面这些影响属于【待验证】，后续用实验结果填实：

这里的关键点是：每个卖点都必须有对应实验，不然就只是“把因素堆进模型”。
### 8.4 解决方案
整体解决思路分成四步。
第一步：网络建模。
把 LEO 网络建模为时变图 G(t) = (V, E(t), X(t))。E(t) 表示随时间变化的链路，X(t) 表示队列、带宽、负载、可靠性、链路剩余时间等状态。
第二步：智能体建模。
每颗卫星是一个智能体。智能体不计算完整端到端路径，只在每次转发时选择下一跳。
第三步：动作过滤。
在 Actor 输出动作前，先过滤不可用链路、低带宽链路、高队列邻居、低可靠链路、快断链路和可能形成环路的动作。
第四步：MAPPO/CTDE 训练与执行。
训练阶段使用集中式 Critic 和全局奖励，执行阶段只部署 Actor 策略网络。每颗卫星根据自身和邻居状态独立选择下一跳。
执行时可以写成下面的步骤：
```text
卫星 i 收到数据包
    ↓
读取自身队列 q_i
    ↓
通过 Hello 包获取邻居队列 q_j、链路负载 rho_ij、可靠性、T_rem_ij
    ↓
动作掩码过滤不可用/高风险下一跳
    ↓
Actor 在候选邻居中选择 j
    ↓
转发数据包，并记录时延、队列、丢包、切换等信息
```
### 8.5 重要意义
方案有效时，意义不只是“路由性能更好”，而是能说明分布式学习型路由在 LEO 场景里有实际价值。
意义
具体体现

### 8.9 回到原点
短期内更适合先在仿真平台中验证，比如自建 LEO 网络仿真环境，或者结合 ns-3、STK 等工具生成拓扑和链路变化。等仿真结果稳定后，再考虑原型平台或更真实的系统验证。
阶段目标不是直接部署到真实卫星，而是先证明三件事：
1. 局部观测够不够用：只用自身和邻居状态，能不能接近或超过全局路由类方法的性能；
2. 队列和链路寿命是否真的有用：去掉这两个因素后，队列、丢包、重路由是否明显变差；
3. 分布式执行是否低开销：Hello 包和 Actor 推理开销是否低于全局状态维护和频繁重计算。
如果这三点成立，这个方向就可以继续往论文实验推进；如果不成立，就需要回到状态设计、奖励函数或 Hello 交换机制重新调整。

## 9. 目前总结
整体思路：
先把 LEO 卫星网络建模为时变图，再参考 RTMDP 把一次路由决策拆成状态、动作、转移和奖励。每颗卫星只利用自身状态、一跳邻居状态和相邻链路状态进行下一跳选择。为了避免拥塞和链路失效，在状态和奖励中加入队列长度、链路负载、链路可靠性和链路剩余可用时间，并通过动作掩码过滤明显不可用或高风险链路。方法上采用 MAPPO/CTDE 思路，实现训练阶段全局协同、执行阶段分布式自治。实验上通过对比实验和消融实验验证该方法在时延、丢包率、负载均衡和路由稳定性方面的效果。
关键逻辑链：
```text
LEO 高动态 + 万颗级规模
        ↓
全局状态维护和集中式最短路开销大
        ↓
只能依赖局部观测和邻居交换
        ↓
局部最短路又容易忽略队列、负载、链路寿命
        ↓
加入队列感知、链路寿命约束和动作掩码
        ↓
用 MAPPO/CTDE 在训练阶段学全局协同，执行阶段保持分布式
        ↓
通过时延、队列、丢包、重路由、控制开销验证是否有效
```
后续需要进一步细化的地方包括：
- 状态空间具体如何归一化和组织；
- 链路剩余可用时间如何计算；
- 奖励函数各项权重如何设置；
- 仿真星座和业务流量如何生成；
- 与已有 LEO 路由算法的公平对比方式。
下一步重点不是继续加概念，而是定下三个东西：
1. 仿真场景参数：星座规模、链路容量、业务到达率、热点流量比例；
2. 奖励函数参数：D_target、Q_max、T_safe 和各项权重；
3. 消融实验表：每个模块去掉后观察哪些指标变化。
这样后续讨论“如何证明”时，可以直接对应到具体实验和指标。
