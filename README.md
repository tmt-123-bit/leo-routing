# 低轨卫星网络分布式动态路由研究思路

题目先收敛到：

**基于 RTMDP 建模的队列感知与链路寿命约束低轨卫星网络分布式动态路由研究**

## 摘要

本文研究的是低轨卫星网络中的分布式动态路由问题。LEO 卫星移动速度快、星间链路频繁变化，且大规模星座下全网链路状态同步开销较高，因此不适合完全依赖地面中心实时计算全局最短路。本文把每颗卫星看作一个智能体，只利用自身状态、一跳邻居状态和相邻链路状态进行下一跳选择。

具体思路是：先把多颗卫星协同转发建成 Dec-POMDP，因为执行时每颗卫星只能看到自身和一跳邻居；再参考 RTMDP 的实时建模方式，把观测过期、动作延迟和链路状态估计误差放进观测设计。在状态中显式加入邻居队列长度、链路负载、链路可靠性和链路剩余可用时间，避免只按传播时延选路导致拥塞或选到快断链路。求解方法上采用 MAPPO/CTDE 思路：训练阶段使用集中式 Critic 和全局奖励学习协同策略，执行阶段每颗卫星只根据局部观测独立选择下一跳。

预期验证重点包括三点：第一，队列和负载感知能否降低最大队列长度和 95% 分位时延；第二，链路寿命约束能否减少丢包、重路由和路由切换；第三，MAPPO/CTDE 的协同训练能否在不显著增加控制开销的情况下，提高投递成功率、吞吐量和负载均衡效果。

## 1. 需要解决的具体问题

低轨卫星网络和普通地面网络不太一样。这个场景里最麻烦的不是“能不能算一条最短路”，而是**这个最短路在高动态和高负载情况下还靠不靠谱**。

场景特点：

- **规模很大**：【文献/公开资料】以 Starlink 为例，Space.com 在 2026 年 6 月上旬的报道中提到，Starlink 活跃卫星数量已经超过 10,500 颗；2026 年 6 月 8 日的发射报道中又提到已超过 10,580 颗。这里主要说明“万颗级星座已经是现实规模”，不把 10,580 当成固定常数。
  参考：https://www.space.com/spacex-starlink-satellites.html

- **移动很快**：【文献/公开资料】低轨卫星维持轨道的速度量级通常在 7.8 km/s 左右，也就是约 28,000 km/h。这个数值会随轨道高度变化，但已经足够说明：星间链路和星地可见关系不能按静态网络理解。
  参考：https://en.wikipedia.org/wiki/Low_Earth_orbit

- **集中式路由开销确实是问题**：【文献结果】Page 等人在 *Distributed Probabilistic Congestion Control in LEO Satellite Networks* 中指出，在密集 LEO 星座中使用集中式最小时延路由会带来显著的信令和计算开销，因此转向基于邻居最新流量信息的分布式拥塞控制。具体开销大小还取决于星座规模、更新周期和状态字段设计。
  参考：https://arxiv.org/abs/2209.08565

- **扁平全局路由在真实轨迹下会频繁更新**：【文献结果】SHORT/MobiCom 2024 在 Starlink、Iridium、Telesat 轨迹上比较稳定分层路由和扁平主动/被动路由，报告 SHORT 相比扁平主动路由节省 9-31,816 次路由表更新和 3-31,908 次全局路由控制开销；同时服务可用性达到 100%，而扁平主动链路状态路由因为慢收敛有 8% 不可用时间。这个结果能支撑“高动态 LEO 中全局/扁平控制面很重”，但它不是拥塞感知路由论文。
  参考：https://doi.org/10.1145/3636534.3649362

- **队列和实时流量变化不能忽略**：【文献结果】2026 年的 *Queue-Aware and Resilient Routing in LEO Satellite Networks Using Multi-Agent Reinforcement Learning* 里提到，传统路由通常假设静态链路指标，不能很好处理 queue backlogs 和实时系统变化；同一篇文章还报告，在它的实验设置中，5 s 重计算间隔下所提方法的开销约为 Dijkstra 的 50%。这个结果只能说明“学习型局部决策有降低开销的可能”，不能直接推广成所有 LEO 场景都一定减半。
  参考：https://arxiv.org/abs/2605.04448

- **热点区域会放大拥塞问题**：【文献结果】Cao 等人在 *Computing-Aware Routing for LEO Satellite Networks* 中提到，热点区域会让可见 LEO 卫星的有限计算/传输资源更加紧张；他们的 computing-aware routing 在该文仿真设置下，相比把原始数据直接下传到地面处理，整体时延最高下降 78.31%。虽然这篇重点是计算感知路由，不是单纯的数据包路由，但它能支撑一个判断：热点区域和资源受限会明显改变路径选择效果。
  参考：https://arxiv.org/abs/2211.08820

本研究的具体问题是：

> 在局部信息受限的条件下，如何让每颗卫星根据邻居队列、链路负载和链路剩余可用时间，分布式地选择下一跳，从而降低端到端时延、丢包率和路由震荡。

重点痛点：

1. **局部信息下的快速路由决策**
   万颗级星座里，如果每次拓扑或流量变化都依赖全局状态收集，再集中计算最短路径，信令和计算开销会很高。已有文献也指出密集 LEO 星座中集中式最小时延路由存在显著开销问题。因此本文只让每颗卫星依赖自身和一跳邻居信息做下一跳选择。

2. **队列积压和链路拥塞**
   最短路径不一定是实际最快路径。如果下一跳队列很长，或者链路已经很忙，即使传播时延低，也可能造成更大的端到端时延。Queue-aware LEO routing 的文献也说明，传统静态链路指标没有充分考虑 queue backlogs 和实时流量变化，这会影响路由效果。

3. **链路即将断开的问题**
   LEO 星间链路具有明显的时间动态性。当前可用的链路可能很快失效，如果路由选择了这种链路，可能导致重路由、丢包和额外时延。因此链路剩余可用时间需要进入动作过滤和奖励设计。

核心不是“最短路怎么求”，而是：

> **在局部信息、实时拥塞和链路高动态同时存在的情况下，怎么让卫星更稳地选下一跳。**

## 2. 整体建模思路

先从网络本身开始建模，再引出多智能体强化学习。这一节不只解释“变量是什么意思”，还要把研究对象、决策变量、约束条件和优化目标说清楚。

常用变量如下：

| 符号 | 直观含义 | 可以理解成 |
|---|---|---|
| `i` | 当前正在做路由决策的卫星 | 现在手里拿着数据包的卫星 |
| `j` | 候选下一跳卫星 | 当前卫星的某个邻居 |
| `q_i` | 当前卫星的队列长度 | 当前卫星自己堵不堵 |
| `q_j` | 下一跳卫星的队列长度 | 准备转发过去的邻居堵不堵 |
| `Q_max` | 队列长度上限 | 超过这个值就认为节点接近拥塞 |
| `bandwidth_ij` | 卫星 `i` 到卫星 `j` 的剩余带宽 | 这条链路还剩多少传输能力 |
| `B_min` | 最低带宽要求 | 低于这个值就不建议继续选 |
| `rho_ij` | 链路负载率 | 这条链路忙不忙 |
| `reliability_ij` | 链路可靠性 | 这条链路成功传输的概率高不高 |
| `R_min` | 最低可靠性要求 | 低于这个值说明链路风险太大 |
| `T_rem_ij` | 链路剩余可用时间 | 这条链路还能维持多久 |
| `T_safe` | 安全通过所需时间阈值 | 链路至少要撑过这段时间才值得选 |

这里的阈值不作为固定结论，而是作为后续实验中的可调参数。比如 `q_j < Q_j^max` 表示下一跳队列不能超过缓存上限，具体拥塞阈值可以写成 `theta_q Q_j^max`；`T_rem_ij > T_safe` 表示链路剩余可用时间要足够数据包安全通过，`T_safe` 需要根据排队、传输、传播和安全余量计算。

### 2.1 时变图建模

LEO 网络拓扑随时间变化，可建模为时变图：

```text
G(t) = (V, E(t), X(t))
```

其中：

- `V` 表示卫星节点集合；
- `E(t)` 表示 t 时刻可用的星间链路；
- `X(t)` 表示节点和链路状态，比如队列、带宽、负载、可靠性、链路剩余可用时间等。

这样处理是因为 LEO 链路随时间变化，不能简单当作静态图。

更规范地说，节点和链路状态可以写成：

```text
X(t) = {x_i(t), x_ij(t) | i ∈ V, (i,j) ∈ E(t)}

x_i(t)  = [q_i(t), Q_i^max, service_i(t)]
x_ij(t) = [d_ij(t), C_ij(t), r_ij(t), rho_ij(t), p_ij^out(t), T_rem_ij(t)]
```

其中 `d_ij(t)` 是传播时延，`C_ij(t)` 是链路容量，`r_ij(t)` 是时间窗口内已占用速率，`rho_ij(t)=r_ij(t)/C_ij(t)` 是链路负载率，`p_ij^out(t)` 是链路中断概率，`T_rem_ij(t)` 是链路剩余可见时间。这样写的好处是：后面状态、奖励和动作过滤都能直接对应到这些变量，不会只停留在概念描述。

“控制开销大”可以用一个数量级例子说明。【估算】假设万颗级星座里有 `N = 10,000` 颗卫星，每颗卫星平均维护 4 条星间链路，那么无向链路数大约是：

```text
|E| ≈ N × 4 / 2 = 20,000
```

如果每条链路状态至少包含：

```text
链路两端 ID、传播时延、剩余带宽、负载率、可靠性、链路剩余时间、时间戳
```

【估算】即使非常粗略地按每条链路 32 B 计算，一次全网链路状态快照也大约是：

```text
20,000 × 32 B ≈ 640 KB
```

这个 640 KB 只是原始链路状态，不包括协议头、转发冗余、可靠广播开销和多跳传播。如果要求每颗卫星都拿到全网状态，那么一次全网同步的总分发量可以达到：

```text
640 KB × 10,000 ≈ 6.4 GB
```

如果每 5 s 更新一次，纯状态分发的数量级就是：

```text
6.4 GB / 5 s ≈ 1.28 GB/s
```

【估算】上面这个计算不是某篇论文的实验值，只是说明数量级：**在万颗级星座里，全网链路状态同步很容易变成较重的控制负担**。【文献结果】2605.04448 那篇 queue-aware MARL 文章也报告，在该文设置的 5 s 重计算间隔下，它的方法开销约为 Dijkstra 的 50%。全局最短路类方法的重计算和状态维护可能成为瓶颈，具体程度需要通过本文仿真实验确认。

### 2.2 智能体建模

每颗卫星看作一个智能体。卫星在转发数据包时，只需要决定当前数据包的下一跳，而不是一次性计算完整路径。

优点：

- 更符合分布式执行；
- 可以根据实时队列和链路状态调整；
- 不需要频繁维护全局端到端路径。

逐跳路由不保证每一步都得到全局最短路径，但更适合高动态场景。提前计算完整路径时，路径中间某条链路断开就需要重算；逐跳路由虽然可能局部绕路，但每一步都能根据当前状态重新判断。

需要验证的不是“每条路径是不是最短”，而是：

```text
逐跳分布式决策能不能在总体上降低丢包、队列积压和链路失效带来的重路由。
```

如果平均跳数略有增加，但 95% 分位时延、丢包率和重路由次数下降，则说明这个取舍是合理的。

从多智能体角度看，智能体集合为：

```text
I = {1, 2, ..., N}
```

智能体 `i` 在时刻 `t` 的决策不是完整路径 `path(s,d)`，而是下一跳动作：

```text
a_i(t) ∈ A_i(t)
```

也就是说，本文的决策粒度是 packet-level 或 flow-level 的逐跳转发。后续实验可以先按 packet-level 建模；如果仿真开销太大，再把同一源目的对的一批数据包聚合成 flow-level 决策。

### 2.3 局部观测设计

每颗卫星能够观测的信息主要包括：

- 自身队列长度；
- 邻居卫星队列长度；
- 相邻链路传播时延；
- 链路剩余带宽；
- 链路负载率；
- 链路可靠性或中断概率；
- 链路剩余可用时间；
- 当前数据包的目的节点或目的区域。

这些状态分别对应：

- 队列长度用于判断下一跳是否拥塞；
- 链路负载用于判断链路是否繁忙；
- 传播时延用于判断基础传输距离；
- 剩余带宽用于判断链路传输能力；
- 可靠性用于判断传输失败风险；
- 链路剩余可用时间用于判断链路是否快要断开。

这里先列的是候选观测量，不等于全部都进入 Actor。针对“变量过多”的问题，本文把它当成状态空间设计问题来处理：变量越多，策略越难训练，星上推理越重，Hello 包也越大。所以本文需要把变量分层，而不是把所有候选因素放进同一个状态向量。

局部候选观测可以先拆成三组：

```text
自身状态：q_i, buffer_i, service_rate_i
邻居状态：q_j, load_j, available_j
链路状态：delay_ij, bandwidth_ij, rho_ij, reliability_ij, T_rem_ij
```

其中最关键的是 `q_j` 和 `T_rem_ij`：

- `q_j` 解决的是“下一跳现在堵不堵”；
- `T_rem_ij` 解决的是“这条链路会不会很快断”。

2605.04448 那篇文章里把单跳时延写成传播时延、传输时延和排队时延的组合，说明队列项会直接进入时延。如果只看传播距离，实际可能选到一条“距离短但队列很长”的路径。

#### 2.3.1 变量取舍原则

本文的取舍原则是：**Actor 只看会直接影响下一跳选择、且星上能低开销获得的变量；其他变量放到动作掩码、训练期 Critic 或评价指标里。** 这样既保留建模严谨性，也避免状态维度无限膨胀。

本文准备用四个判据筛变量：

| 判据 | 具体判断 | 不满足时怎么处理 |
|---|---|---|
| 局部可观测性 | 卫星 `i` 能否通过本地测量或一跳 Hello 包得到 | 不能放进执行期 Actor，只能放进训练期 Critic 或评价指标 |
| 因果相关性 | 该变量是否直接改变下一跳的时延、丢包、拥塞或链路中断风险 | 如果只是结果统计，就不作为状态输入 |
| 非冗余性 | 是否和已有变量表达同一件事，例如 `bandwidth_ij` 与 `rho_ij` | 初版只保留一个，另一个放入掩码或消融实验 |
| 获取成本 | 是否会显著增加 Hello 字段长度、测量成本或推理维度 | 成本高但收益不确定的变量先不进主模型 |

这四条主要回答“变量太多时怎么取舍”：不是因为某个因素听起来重要就加入，而是要能说明它在执行期能不能拿到、对下一跳有没有直接作用、和已有变量是不是重复、加入后收益能不能覆盖开销。

具体分成四层：

| 层次 | 放哪些变量 | 为什么放这里 | 不直接放进 Actor 的原因 |
|---|---|---|---|
| Actor 核心输入 | 自身队列 `q_i`、邻居队列 `q_j`、传播时延 `delay_ij`、链路负载 `rho_ij`、可靠性 `reliability_ij`、剩余寿命 `T_rem_ij`、信息年龄 `age_j`、到目的方向/进展 `progress_ij` | 这些变量会直接改变“下一跳 j 值不值得选” | 保持执行时输入小，便于星上部署 |
| 动作掩码/硬约束 | 链路是否可用、剩余带宽是否满足业务、是否成环、`T_rem_ij > T_safe`、可靠性是否低于安全线 | 这些属于可行性约束，不应该让强化学习慢慢试错 | 不作为连续状态学习，而是先过滤不可行动作 |
| 训练期 Critic/全局奖励 | 全网队列统计、全网丢包、投递成功、`J_load`、重路由次数、路由切换次数 | 训练时用来让策略朝全局目标优化 | 星上执行时单颗卫星拿不到全网状态 |
| 评价指标 | 吞吐量、P95/P99 时延、流完成率、多流高负载完成情况、控制开销、Actor 推理时间、模型参数量 | 用来证明方法是否有效和是否可部署 | 这些是实验结果，不是单跳转发时能直接观测的状态 |

这样做以后，最终 Actor 的单邻居输入可以写成一个较小的向量：

```text
z_ij(t) = [
  q_j(t) / Q_j^max,
  delay_ij(t) / D_ref,
  rho_ij(t),
  reliability_ij(t),
  T_rem_ij(t) / T_safe,
  age_j(t) / T_hello,
  progress_ij(t)
]

o_i(t) = [q_i(t) / Q_i^max, {z_ij(t)}_{j ∈ N_i(t)}, dst]
```

这里每一项都做无量纲化，避免不同量纲的数值直接混在一起：

| 特征 | 归一化方式 | 作用 |
|---|---|---|
| `q_j/Q_j^max` | 队列占用率 | 判断下一跳是否拥塞 |
| `delay_ij/D_ref` | 单跳时延相对参考值 | 防止过度绕路 |
| `rho_ij` | 本身就在 `[0,1]` | 判断链路是否繁忙 |
| `reliability_ij` | 滑动窗口成功率或 `1-P_out` | 判断传输失败风险 |
| `T_rem_ij/T_safe` | 可再做截断，例如超过安全时间很多时统一记为安全 | 判断链路是不是快断 |
| `age_j/T_hello` | 邻居信息的新鲜程度 | 判断 Hello 状态是否过期 |
| `progress_ij` | 目的方向进展，可由距离差归一化 | 防止只避开拥塞而一直绕远 |

其中 `progress_ij(t)` 表示选择邻居 `j` 后是否更接近目的区域，可以由几何距离、轨道坐标距离或分层路由域距离给出。一个可写法是：

```text
progress_ij(t) = (dist(i,dst) - dist(j,dst)) / D_ref
```

`progress_ij(t) > 0` 表示下一跳更接近目的地，`progress_ij(t) < 0` 表示在绕远。这个变量的作用是防止策略只顾避开拥塞，结果一直绕远。

如果每个邻居特征维度为 `d_z = 7`，当前候选邻居数为 `k_i(t)=|N_i(t)|`，则执行期输入维度大致是：

```text
d_o(t) = d_self + k_i(t) * d_z + d_dst
```

在初始 4 条 ISL 场景里，`k_i(t)` 通常较小，所以这个输入规模还能接受；如果后续加入地面站链路、机会链路或壳间链路，`k_i(t)` 变大，就不能简单把所有邻居拼成很长的固定向量。更稳的做法是对每个 `z_ij(t)` 使用共享邻居编码器，再通过动作掩码在候选邻居上做选择：

```text
h_ij = f_theta(z_ij)
π(a_i=j | o_i) = masked_softmax(g_theta(h_ij, q_i, dst))
```

这样参数量不会随邻居数量线性增加，只是候选动作数量变多；也更符合分布式星上执行的计算约束。后续如果用 GNN 或 attention，也可以把这里的共享邻居编码器换成图编码器，但第一版先不引入更复杂结构，避免变量和模型复杂度一起膨胀。

有些变量看起来重要，但暂时不把它们作为 Actor 核心输入：

| 暂不进入 Actor 的变量 | 处理方式 | 原因 |
|---|---|---|
| 剩余带宽 `bandwidth_ij` | 主要用于动作掩码，或与 `rho_ij` 二选一 | `bandwidth_ij` 和 `rho_ij` 都反映链路忙闲，同时放入容易冗余；初期保留 `rho_ij` 更直观 |
| 链路中断概率 `P_out` | 与 `reliability_ij` 二选一 | 二者本质上都反映失败风险，先用滑动窗口可靠性，后续再做替换实验 |
| 吞吐量 | 作为评价指标 | 吞吐量是全局统计结果，不是单跳邻居状态 |
| 流完成率 | 作为评价指标，先不作为主奖励 | SHORT 只报告 service availability，不支持直接推出 flow completion rate，本文需要自己实验 |
| P95/P99 尾时延 | 作为评价指标或全局奖励的扩展项 | 单跳执行时无法直接知道全局尾时延，先用队列和局部时延间接控制 |
| 星上 CPU/内存 | 作为部署开销指标 | 它约束模型大小，但不是路由状态 |
| SNR、BER、能量、计算负载 | 暂不纳入主模型 | 会显著增加状态维度；除非实验发现可靠性或资源瓶颈无法解释结果，再逐步加入 |

所以变量不是越多越好。本文第一版模型先保留“队列、时延、负载、可靠性、链路寿命、信息年龄、目的方向”这几个最直接影响下一跳的变量；后续通过消融实验判断每个变量是否真的有贡献。如果某个变量去掉后指标几乎不变，就应从 Actor 输入里删掉，只保留为评价指标或备用分析变量。

变量消融可以按下面顺序做，而不是一次性比较所有组合：

| 版本 | Actor 输入 | 想验证什么 |
|---|---|---|
| V0 最小状态 | `delay_ij + progress_ij` | 只看几何/时延时的基础性能 |
| V1 队列感知 | V0 + `q_j` | 队列是否改善热点拥塞和尾时延 |
| V2 负载感知 | V1 + `rho_ij` | 链路负载是否改善 Jain 公平性和最大链路利用率 |
| V3 稳定性感知 | V2 + `reliability_ij + T_rem_ij` | 可靠性和链路寿命是否减少丢包、重路由 |
| V4 实时增强 | V3 + `age_j + a_i^{t-1}` | 信息过期和动作延迟建模是否有用 |

如果从 V3 到 V4 的收益很小，但推理时间和 Hello 开销明显增加，就说明实时增强变量在当前仿真设置下可以简化；反过来，如果高动态场景下 V4 明显降低丢包和路由切换，就说明这些变量值得保留。

每个观测量对应的全局影响如下：

| 因素 | 直接影响 | 对全局指标的影响 | 怎么计算/验证 |
|---|---|---|---|
| 自身队列 `q_i` | 当前节点排队压力 | 影响本节点后续转发时延和丢包 | 记录当前缓存中等待转发的数据包数或字节数，使用 `q_i/Q_i^max` 做归一化；不预设固定阈值，后续画队列占用率与时延/丢包的关系曲线 |
| 邻居队列 `q_j` | 下一跳等待时间 | 影响端到端时延、最大队列、热点拥塞 | `q_j` 来自邻居 Hello 包；“去掉 `q_j`”指消融实验中不把邻居队列放入状态、奖励和过滤条件，再比较最大队列和 95% 分位时延 |
| 传播时延 `delay_ij` | 单跳基础时延 | 影响平均端到端时延 | 可由卫星间距离除以光速近似得到，也可由仿真器直接给出；与最短路径平均时延对比，观察是否过度绕路 |
| 剩余带宽 `bandwidth_ij` | 链路传输能力 | 影响吞吐量和链路排队 | `bandwidth_ij = C_ij - r_ij`，其中 `C_ij` 是链路容量，`r_ij` 是时间窗口内已占用速率；通过不同负载水平下的吞吐和排队变化验证 |
| 链路负载 `rho_ij` | 链路繁忙程度 | 影响负载均衡和热点链路 | `rho_ij = r_ij / C_ij`；重点看最大链路利用率、Jain 负载公平性指数和 95% 分位时延，而不是只给一个固定阈值 |
| 可靠性 `reliability_ij` | 传输失败风险 | 影响丢包率和重传/重路由 | 可由滑动窗口内成功传输比例估计，也可由物理层中断概率得到：`reliability_ij = 1 - P_out(i,j)`；不预设固定过滤阈值，后续通过故障场景和敏感性实验确定 |
| 剩余可用时间 `T_rem_ij` | 链路是否快断 | 影响路径中断、重路由、丢包 | 由轨道/拓扑预测得到当前链路剩余可见时间；比较加入和去掉 `T_rem` 后的路径中断次数、丢包率和重路由次数 |

状态是否有用，主要通过消融实验判断：去掉某个状态后，观察对应全局指标是否变差。

链路可靠性的计算可以参考 ETX 思路。De Couto 等人在 MobiCom 2003 中提出 ETX 指标，用前向投递率 `D_f` 和反向 ACK 成功率 `D_r` 估计一条链路的期望传输次数：

```text
ETX = 1 / (D_f * D_r)
```

本文里的 `reliability_ij = successful_transmissions / total_attempts` 可以理解成 ETX 中投递率思想的简化版本。也就是说，这里的“可靠性”不是凭空设定，而是用滑动窗口内的实际收发成功情况估计链路质量。

负载均衡度后续不只用方差或变异系数，因为这两个数不太直观。可以优先用 Jain 公平性指数：

```text
J_load = (sum_e rho_e)^2 / (|E| * sum_e rho_e^2)
```

`J_load` 越接近 1，说明负载越均匀；越低说明流量越集中在少数链路上。这里不预设某个方差阈值一定正确，而是看完整方法能不能把 `J_load` 往 1 的方向推，同时不牺牲时延和投递成功率。

### 2.4 部分可观测建模

这里需要把“全局真实状态”和“卫星实际能看到的状态”分开。训练时仿真器可以看到全局状态 `S(t)`，但星上执行时卫星 `i` 只能看到局部观测 `o_i(t)`：

```text
S(t) = {X(t), packet states, topology states}

o_i(t) = [q_i(t)/Q_i^max, {z_ij(t)}_{j ∈ N_i(t)}, dst]
```

其中 `N_i(t)` 是卫星 `i` 当前的一跳候选邻居集合，`dst` 是当前数据包目的节点或目的区域，`z_ij(t)` 是上一节定义的单邻居紧凑特征。这样写是为了避免把全网状态或大量冗余链路字段塞进执行期 Actor。

所以本文的主问题更接近 Dec-POMDP，而不是普通 MDP。普通 MDP 默认一个智能体能看到完整状态；但 LEO 路由里是很多颗卫星一起转发，每颗只看得到自己和一跳邻居，这正是“多智能体 + 部分可观测”的问题。可以写成：

```text
Dec-POMDP = <I, S, {A_i}, P, R, {Ω_i}, O, gamma>
```

含义如下：

| 元素 | 本文含义 |
|---|---|
| `I` | 卫星智能体集合 |
| `S` | 全局网络状态，训练时 Critic 可见 |
| `{A_i}` | 卫星 `i` 的动作空间，执行时在时变候选集 `A_i(t)` 中选下一跳 |
| `P` | 联合状态转移 `P(s'|s,a)`，包括队列、链路、拓扑和包位置变化 |
| `R` | 协作型全局共享奖励，对应后面的 `R_global` |
| `Ω_i` | 卫星 `i` 的局部观测空间，即自身、一跳邻居和相邻链路 |
| `O` | 观测函数 `O(o_i|s)`，把全局真状态映射成局部、延迟、可能有误差的观测 |
| `gamma` | 折扣因子 |

这里的关键是 `S` 和 `o_i` 不一样：`S` 是全网真实状态，星上执行时拿不到；`o_i` 是卫星通过本地测量和 Hello 包得到的局部信息。后面再用 RTMDP 思路补上“实时性”，也就是决策延迟、观测过期和估计误差。

### 2.5 动作设计

动作就是选择下一跳：

```text
a_i(t) = 选择某个邻居卫星作为下一跳
```

这里的“邻居”不应固定理解成四个前后/左右 ISL。更一般地，当前卫星 `i` 在时刻 `t` 的候选动作集合可以写成：

```text
A_i(t) = { j | (i, j) ∈ E(t), j 是当前可达的候选下一跳 }
```

候选下一跳可以包括：

- 同轨/异轨 ISL 邻居；
- 当前可见的地面站链路；
- 可用的壳间链路或机会捷径链路；
- 分层路由中允许进入的下一层路由域。

实验初期可以先从规则网格星座的 4 条 ISL 做起，因为这样仿真可控；但建模上要保留 `A_i(t)` 这个时变候选集合，否则后面很难扩展到更复杂的链路结构。

### 2.6 动作过滤

真正选择下一跳之前，先对 `A_i(t)` 做一层动作过滤，把明显不适合的链路排除掉：

- 链路当前不可用；
- 剩余带宽不足；
- 下一跳队列过长；
- 链路可靠性过低；
- 链路剩余可用时间太短；
- 可能形成路由环路。

强化学习不需要从零学习这些硬约束。先过滤不可行动作，可以减少无效探索，也能提高路由稳定性。

动作过滤可以写成下面这种规则：

候选下一跳需要满足六个条件：

1. 这条链路现在真的能用；
2. 这条链路剩余带宽不能太低；
3. 下一跳卫星不能已经排了很长队；
4. 这条链路不能太不可靠；
5. 这条链路不能马上就断；
6. 这个下一跳不能让数据包绕回已经走过的节点。

对应成符号就是：

```text
候选下一跳 j ∈ A_i(t) 必须同时满足：
1. link_ij 当前可用
2. bandwidth_ij > B_min
3. q_j < Q_max
4. reliability_ij > R_min
5. T_rem_ij > T_safe
6. j 不在当前数据包已访问节点集合中
```

其中 `T_safe` 不应随意设为常数，至少要大于一次转发可能需要的时间：

```text
T_safe > T_queue + T_transmission + T_propagation + T_margin
```

如果链路剩余可用时间不足以支撑当前数据包安全通过，即使它当前可用，也不应该选。

这一点区别于直接让策略网络在所有邻居里选的做法：先用网络约束做 action mask，再让策略在可行动作里选择。

动作过滤条件及其全局影响如下。表中的“影响大小”属于【待验证】，需要靠消融实验给出具体数值：

| 过滤条件 | 不过滤会怎样 | 影响的全局指标 | 预期影响大小怎么观察 |
|---|---|---|---|
| `link_ij` 当前可用 | 可能选择物理上不可转发的链路 | 丢包率、无效动作比例 | 无效动作比例应接近 0 |
| `bandwidth_ij > B_min` | 低带宽链路被继续塞流量 | 传输时延、吞吐量、队列长度 | 去掉后高负载下吞吐下降、队列上升 |
| `q_j < Q_max` | 把包推给已拥塞邻居 | 最大队列、95% 分位时延、丢包率 | 去掉后最大队列明显上升 |
| `reliability_ij > R_min` | 选择高失败风险链路 | 丢包率、重路由次数 | 去掉后故障场景丢包率上升 |
| `T_rem_ij > T_safe` | 选择快断链路 | 路径中断次数、路由切换次数、丢包率 | 去掉后链路频繁断开场景重路由增加 |
| 不在已访问节点集合中 | 可能形成路由环路 | TTL 超时丢包、平均跳数、链路占用 | 去掉后平均跳数和 TTL 丢包上升 |

影响大小通过消融实验量化。例如：

```text
完整动作过滤 vs 去掉 T_rem 过滤
观察：路径中断次数、丢包率、路由切换次数
```

如果去掉 `T_rem` 过滤后，平均时延变化不大，但路由切换次数和链路断开导致的丢包明显上升，则说明链路寿命主要影响稳定性，而不是平均时延。

### 2.7 优化目标

最后要把建模落到一个优化问题上。本文的目标不是单独最小化某一条路径的跳数，而是在局部观测和低开销约束下优化全网 QoS。可以写成：

```text
minimize    alpha_1 * D_e2e
          + alpha_2 * Q_cong
          + alpha_3 * P_loss
          + alpha_4 * Reroute
          + alpha_5 * (1 - J_load)

subject to  a_i(t) ∈ A_i^valid(t)
            overhead_ctrl <= B_ctrl
            T_rem_ij(t) > T_safe, if link (i,j) is selected
```

其中：

| 项 | 含义 |
|---|---|
| `D_e2e` | 平均或分位端到端时延 |
| `Q_cong` | 全网平均队列、最大队列或队列占用率 |
| `P_loss` | 丢包率 |
| `Reroute` | 路由切换、路径中断或重路由次数 |
| `J_load` | Jain 负载公平性指数，越接近 1 表示链路负载越均衡 |
| `A_i^valid(t)` | 经过动作掩码后的可行动作集合 |
| `overhead_ctrl` | Hello 包和状态交换带来的控制开销 |
| `B_ctrl` | 允许的控制开销预算 |

如果后续实验还没做通信预算优化，可以先把 `overhead_ctrl <= B_ctrl` 写成约束指标，而不是奖励项。这样比较稳：主方法先解决路由性能，实验里再证明控制开销没有超过可接受范围。

为了更接近 IEEE 论文里的建模写法，这里还需要把“优化目标”和“学习目标”区分开：

```text
论文里的系统目标：在局部观测和控制开销约束下，降低全网时延、拥塞、丢包和重路由。
强化学习里的训练目标：最大化折扣累计全局奖励 E[sum_t gamma^t R_global(t)]。
```

二者不是两套不同目标，而是同一个问题的两种表达。系统目标说明网络最终想要什么，强化学习目标说明 MAPPO/CTDE 在训练时怎么逼近这个目标。

本文的建模闭环可以整理成下面这张表：

| 建模环节 | 本文具体定义 | 需要避免的问题 |
|---|---|---|
| 网络模型 | 时变图 `G(t)=(V,E(t),X(t))` | 不能把 LEO 当成静态图 |
| 智能体 | 每颗卫星是一个 agent | 不能只做单节点最优 |
| 局部观测 | `o_i(t)=[q_i,{z_ij},dst]` | 不能让执行期 Actor 偷看全网状态 |
| 动作空间 | `A_i(t)` 是当前可达下一跳集合 | 不能假设动作集合永远固定 |
| 动作约束 | `A_i^valid(t)` 由链路可用性、带宽、可靠性、链路寿命和环路过滤得到 | 不能让 RL 通过大量失败样本学习硬约束 |
| 状态转移 | 队列、链路、拓扑和包位置随联合动作变化 | 不能只写奖励，不写动作怎么改变系统 |
| 全局目标 | 时延、拥塞、丢包、重路由、负载均衡和控制开销 | 不能只优化平均最短路 |
| 求解方式 | MAPPO/CTDE，训练用全局 Critic，执行只用本地 Actor | 不能部署时依赖地面实时全局控制 |

这张表后面可以直接扩成论文的 System Model 和 Problem Formulation 两节。

## 3. 方法选择：实时 Dec-POMDP 建模 + MAPPO/CTDE 求解

这一节不再把方法简单写成“RTMDP + MAPPO”，因为那样没有交代多卫星协作和局部可观测到底放在哪里。更合适的说法是分成三层：

```text
第一层：Dec-POMDP，说明这是多卫星协作 + 局部可观测问题
第二层：RTMDP 实时增强，说明观测过期、动作延迟和估计误差怎么进入模型
第三层：MAPPO/CTDE，说明训练和执行时具体怎么求解
```

这样写以后，RTMDP 不再承担整个问题模型。主模型是 Dec-POMDP，RTMDP 只是把“实时决策”这部分补进去。

### 3.1 实时 Dec-POMDP 怎么落到本文

Dec-POMDP 负责回答“这个问题本质是什么”。对应到本文：

| Dec-POMDP 元素 | 本文对应内容 | 解决的问题 |
|---|---|---|
| 多智能体 `I` | 所有卫星都是智能体 | 不是单颗卫星自己优化 |
| 全局状态 `S` | 全网队列、链路、拓扑和数据包位置 | 训练时可以用来评价全局效果 |
| 局部观测 `o_i` | 卫星 `i` 的自身状态、一跳邻居和相邻链路 | 执行时不能依赖全网状态 |
| 动作 `a_i` | 在候选集合 `A_i(t)` 中选下一跳 | 路由动作就是逐跳转发 |
| 共享奖励 `R_global` | 投递成功、时延、队列、丢包等综合目标 | 避免每颗卫星只追求局部最短 |

RTMDP 负责回答“这个问题为什么还要考虑实时性”。LEO 场景里，卫星做动作时会遇到三个实时问题：

| 实时问题 | 在 LEO 路由里的表现 | 本文怎么处理 |
|---|---|---|
| 决策延迟 | Actor 推理、排队和传播都会让动作不是瞬间生效 | 把上一时刻已经算出但正在生效的动作 `a_i^{t-1}` 放入增广观测 |
| 观测时效 | Hello 包里的邻居队列/负载来自过去某个时刻 | 给邻居状态加 `age_j(t)`，表示这条信息已经过了多久 |
| 观测误差 | `reliability_ij`、`T_rem_ij` 都是估计量 | 把观测函数 `O(o_i|s)` 看成随机映射，并做估计误差实验 |

所以卫星 `i` 真正给 Actor 的输入不是单纯的 `o_i^t`，而是增广观测。考虑到 `age_j(t)` 已经在 `z_ij(t)` 里出现，最终可以写成：

```text
õ_i^t = (o_i^t, a_i^{t-1})
```

这里 `a_i^{t-1}` 的作用是告诉策略“上一步的动作还可能影响现在的队列和包位置”；`age_j(t)` 的作用是告诉策略“这个邻居状态新不新”。如果不加入这些量，模型默认观测实时准确，但这和 LEO 高动态场景不太一致。

对应到一次转发决策，可以写成：

```text
输入：增广局部观测 õ_i^t、目的节点 dst、动作掩码 mask_i(t)
动作：a_i^t ∈ A_i^valid(t)，即选择一个可行下一跳
转移：数据包位置改变，队列/链路/拓扑继续随时间变化
反馈：由局部代价和全局奖励共同评价这次选择是否好
```

这样，Dec-POMDP 负责“多智能体局部观测”，RTMDP 负责“实时状态不完美”，两者合起来才是本文的建模思路。

### 3.2 为什么用 MAPPO/CTDE 求解

1. **这是一个多智能体问题**
   每颗卫星的转发决策都会影响其他卫星的队列和链路负载，所以不能只看单个节点的局部最优。

2. **适合 CTDE 思路**
   训练阶段可以在地面仿真平台获取全局状态，所以可以使用集中式 Critic；但真正执行时，每颗卫星只能用本地观测，所以只部署 Actor。

3. **卫星节点比较同构**
   大多数卫星的转发功能类似，因此 Actor 可以采用统一结构，后续再根据星上资源决定是否压缩模型。

4. **动作是离散的**
   路由动作就是从邻居中选择一个下一跳，比较适合 MAPPO 这种离散动作策略方法。

5. **方便加入动作掩码**
   LEO 网络中邻居集合会变化，部分链路也会不可用，所以动作掩码比较重要。

更具体地说，本文采用共享 Actor + 集中式 Critic：

```text
Actor:  π_theta(a_i | õ_i^t, mask_i(t))
Critic: V_phi(S^t) 或 Q_phi(S^t, a^t)
```

Actor 的输入只包含星上能拿到的增广局部观测和动作掩码，所以执行阶段可分布式部署；Critic 在训练时可以看到全局状态，用来学习多个卫星动作对全网指标的影响。

训练阶段：

- Critic 可以看到全局网络状态；
- 用全局奖励引导多颗卫星协同；
- 目标是降低全网时延、队列积压、丢包率和路由切换。

执行阶段：

- 每颗卫星只使用 Actor；
- 输入是局部观测和邻居状态；
- 输出是下一跳选择；
- 不依赖地面中心实时控制。

训练和执行阶段的信息差异可以写成：

| 阶段 | 谁可见全局状态 | 用什么网络 | 目标 |
|---|---|---|---|
| 训练 | 地面仿真平台和集中式 Critic 可见 `S(t)` | Actor + Critic | 学到能改善全局奖励的协同策略 |
| 执行 | 单颗卫星不可见全局状态 | 只部署 Actor | 根据 `õ_i^t` 和动作掩码独立选下一跳 |

和其他方法的对比如下：

| 方法 | 主要问题 | 为什么不作为主方案 |
|---|---|---|
| Dijkstra | 依赖较新的全局链路状态，主要优化最短路径 | 高动态和高负载下容易忽略队列、链路寿命和状态更新开销 |
| Q-routing | 分布式、轻量 | 表格型方法表达能力有限，难同时处理队列、负载、可靠性、链路寿命等连续状态 |
| DQN | 可以处理复杂状态 | 更偏单智能体，难直接表达多颗卫星之间的协作关系 |
| QMIX | 适合合作 MARL | 动作空间通常要求更规整，而 LEO 邻居集合会随时间变化，动作掩码处理不如 MAPPO 直接 |
| MAPPO/CTDE | 能做多智能体协作 | 普通 MAPPO 还不够，需要结合 Dec-POMDP 局部观测、RTMDP 实时增强、队列/链路寿命奖励和动作掩码 |

这里说的 MAPPO/CTDE 不是再单独起一个新名字，而是使用标准 MAPPO 思路做求解，再把本文自己的部分放在状态、动作掩码和奖励里：

| 组成 | 采用已有方法 | 本文需要自己设计的部分 |
|---|---|---|
| 多智能体训练框架 | MAPPO/CTDE | 训练时 Critic 看全局状态，执行时 Actor 只看局部状态 |
| 策略网络输入 | 局部观测 Actor | `z_ij(t)` 的队列、负载、可靠性、链路寿命和信息年龄设计 |
| 动作选择 | 离散动作策略 | 时变候选动作集 `A_i(t)` 和动作掩码 `mask_i(t)` |
| 协作目标 | 共享全局奖励 | 队列、时延、丢包、链路寿命和负载均衡之间的权衡 |
| 实时性 | RTMDP/实时增强思想 | 把 `age_j(t)`、`a_i^{t-1}` 和估计误差放入建模 |

所以论文里可以这样说：本文不提出一个全新的 MAPPO 变体名称，而是提出一个面向 LEO 分布式路由的实时 Dec-POMDP 建模和 MAPPO/CTDE 求解框架。创新点落在路由状态设计、约束化动作空间、队列/链路寿命奖励和变量取舍，而不是给 MAPPO 改名。

本文方法可以概括为：

```text
Dec-POMDP 多卫星局部观测建模
    +
RTMDP 实时观测增强
    +
MAPPO/CTDE 协同训练
    +
队列感知、链路寿命约束和动作掩码
```

【待验证】这个选择需要实验验证。比如比较：

```text
普通 MAPPO vs Dec-POMDP/RTMDP 增强观测 + 动作掩码 + 队列/链路寿命奖励
```

若加入动作掩码后无效动作比例下降、训练收敛更快、丢包率下降，则说明该设计有效。

## 4. 奖励函数设计思路

奖励函数不宜把所有指标都直接加进去。吞吐、投递成功率、丢包率、队列长度、时延之间本身存在相关关系，如果全部放入奖励，可能导致重复计分或权重难以解释。

奖励分成两层：

```text
训练阶段：全局协同奖励 R_global
单跳选择：局部即时奖励 r_local
```

全局奖励解决“大家别只顾自己”的问题；局部奖励解决“这一跳到底选哪个邻居”的问题。

### 4.1 全局协同奖励

全局奖励先保留最关键的四类目标：投递成功、端到端时延、拥塞程度、链路失效/丢包。吞吐量、控制开销、路由切换、流完成率和 P95/P99 尾时延先作为评价指标和约束观察，不一开始全部写进奖励。

这里的取舍逻辑是：奖励项太少，策略可能学不到拥塞和失效风险；奖励项太多，权重会很难解释，而且不同指标之间会重复计分。比如吞吐量、投递成功率和丢包率高度相关，队列长度和尾时延也高度相关。所以第一版奖励只保留最能解释路由行为的核心项，其余指标用实验结果报告。

```text
R_global =
  + w_S * S_delivery
  - w_L * C_delay
  - w_Q * C_queue
  - w_P * C_loss
```

`S` 表示 score，越大越好；`C` 表示 cost，越小越好。`w` 表示权重，例如 `w_P` 越大，表示越不能接受丢包；`w_Q` 越大，表示越希望避免队列积压。

直观上：

```text
奖励 = 好事情的加分 - 坏事情的扣分
```

好事情主要是包能送达；坏事情主要是时延高、队列长和丢包。

| 奖励项 | 怎么算 | 为什么保留 | 对奖励的影响 |
|---|---|---|---|
| `S_delivery` | `delivered_packets / generated_packets` | 路由首先要保证包能送达 | 正奖励，越大越好 |
| `C_delay` | `avg_delay / D_target` 或分位时延归一化 | 体现端到端 QoS | 惩罚，越小越好 |
| `C_queue` | `avg_queue/Q_max` 和 `max_queue/Q_max` 的组合 | 直接反映拥塞和热点队列 | 惩罚，越小越好 |
| `C_loss` | `dropped_packets / generated_packets` | 反映拥塞、链路失效和环路导致的失败 | 强惩罚 |

其他指标不删除，但先不直接进入主奖励：

| 指标 | 暂时作为 | 原因 |
|---|---|---|
| 吞吐量 | 评价指标 | 与投递成功率和负载有关，直接放入奖励可能重复计分 |
| Jain 负载公平性指数 | 评价指标/辅助奖励 | 与队列和时延相关，先通过消融判断是否需要单独加入 |
| P95/P99 尾时延 | 评价指标/后续扩展奖励 | 尾时延很重要，但单跳奖励很难直接稳定优化；先看队列项能否间接改善 |
| 流完成率/多流完成情况 | 评价指标 | SHORT 没有报告 flow completion rate，本文不能借 SHORT 支撑，必须靠自己的多流实验验证 |
| 路由切换率 | 评价指标/约束 | 频繁切换会影响稳定性，但过强惩罚可能妨碍绕开拥塞 |
| 控制开销 | 约束指标 | 主要由 Hello 周期和状态字段决定，不一定适合作为每步奖励 |

权重不直接定死，先按重要性排序：

```text
投递成功率/丢包  >  时延  >  队列拥塞
```

【实验前假设】初始权重只作为调参起点，不是文献结论：

```text
w_S = 2.0   投递成功率
w_P = 2.0   丢包惩罚
w_L = 1.5   时延惩罚
w_Q = 1.0   队列惩罚
```

这样设置的原因是：如果包送不到，吞吐或时延再好也没有意义，所以投递成功率和丢包权重要更高。负载均衡、路由切换和控制开销先放到评价指标里，等实验发现主奖励不足时再考虑加入辅助项。

正式实验里不应只报告这一组权重，而要做权重敏感性。更像论文的写法是：先把所有代价项归一化到接近 `[0,1]` 的量级，再在几组权重下观察趋势是否稳定。如果只有某一组权重表现好，说明方法不稳；如果多组权重下都能降低丢包、队列和尾时延，说明建模更可信。

【待验证】权重敏感性实验：

- 增大 `w_Q`，看队列是否下降，但路径是否绕远；
- 增大 `w_L`，看平均时延是否下降，但是否更容易把流量集中到短路径；
- 增大 `w_P`，看丢包率是否下降，但是否牺牲时延或吞吐；
- 加入/不加入负载均衡辅助项，比较最大链路利用率和 Jain 负载公平性指数。

【待验证】全局奖励的作用通过实验量化：

```text
完整方法 vs 去掉全局协同奖励
```

如果去掉全局奖励后，平均时延变化不大，但最大队列长度上升、Jain 负载公平性指数下降、丢包率上升，则说明全局奖励主要作用在削峰和负载均衡，而不是只优化平均值。

### 4.2 局部即时奖励

局部即时奖励主要负责某一跳怎么选。【实验前假设】一个初步形式可以写成：

```text
r_local(i, j) =
  + eta_1 * I_deliver
  - eta_2 * delay_ij_norm
  - eta_3 * queue_j_norm
  - eta_4 * risk_ij
  - eta_5 * lifetime_penalty_ij
  - eta_6 * I_loop
```

每一项对应的意思如下。这里不直接给固定危险阈值，先说明计算方式，阈值后续通过敏感性实验确定。

| 项 | 怎么算 | 怎么验证 | 影响 |
|---|---|---|---|
| `delay_ij_norm` | 单跳传播+传输时延除以场景内参考时延 | 与最短路径基线比较，观察是否过度绕路 | 抑制过远链路 |
| `queue_j_norm` | `q_j / Q_j^max` | 做去掉邻居队列的消融，比较最大队列和 95% 分位时延 | 避免队列长的邻居 |
| `risk_ij` | `1 - reliability_ij` 或物理层中断概率 `P_out(i,j)` | 在链路故障场景下比较丢包率和重路由次数 | 降低丢包 |
| `lifetime_penalty_ij` | 与 `T_safe / T_rem_ij` 相关 | 比较加入/去掉 `T_rem` 后的路径中断次数 | 避免快断链路 |
| `I_loop` | 形成环路为 1 | 统计 TTL 超时丢包和平均跳数 | 避免循环转发 |

链路剩余时间采用更硬的约束：

```text
if T_rem_ij < T_safe:
    直接 mask 掉
else:
    lifetime_penalty_ij = T_safe / T_rem_ij
```

下面只是说明惩罚函数的形状，具体 `T_safe` 需要由仿真场景里的排队、传输、传播和安全余量计算：

```text
若 T_safe = 2 s:
T_rem = 10 s  -> penalty = 0.2
T_rem = 3 s   -> penalty = 0.67
T_rem = 1 s   -> mask 掉
```

即：**剩余时间低于安全阈值直接过滤，高于阈值但越接近阈值惩罚越大。**

局部奖励的目标不是让每一跳都选最短，而是在下面四个因素之间折中：

```text
短路径 + 低队列 + 低负载 + 不快断
```

因此算法不一定选择传播时延最小的邻居；如果该邻居队列很长或链路快断，端到端结果反而可能更差。

## 5. 预期创新点

可以提炼成三个创新点：

1. **队列感知的分布式路由决策**
   在下一跳选择中加入邻居队列长度，避免流量集中到局部最短路径。和已有 queue-aware 路由的区别不能停在“也使用了队列”，而是把队列放进 Dec-POMDP 的局部观测和 RTMDP 增强观测中，重点看只交换邻居信息时，队列状态还能不能改善热点拥塞和尾时延。

2. **面向时变候选动作集合的约束化决策**
   动作不是固定四邻居，而是当前时刻的候选集合 `A_i(t)`。在策略网络选择动作前，先过滤不可用、低可靠、快断或可能成环的链路，减少无效动作和路径中断。这个部分更像工程约束和实时路由机制，不单独夸成算法创新。

3. **基于实时 Dec-POMDP + MAPPO/CTDE 的协同路由框架**
   先用 Dec-POMDP 表达多卫星协作和局部可观测，再用 RTMDP 思路补上观测时效、决策延迟和估计误差，最后用 MAPPO/CTDE 在训练阶段学习协同策略，执行阶段每颗卫星只根据局部信息独立决策。

每个创新点对应一组验证实验：

| 创新点 | 需要证明什么 | 对应实验 |
|---|---|---|
| 队列感知 | 在只交换邻居信息的条件下，队列状态是否仍能缓解热点拥塞 | 高负载热点场景，比较平均队列、最大队列、95% 分位时延 |
| 约束化动作集合 | 动作过滤是否减少无效动作和快断链路导致的失败 | 链路频繁断开场景，比较无效动作比例、丢包率、重路由次数 |
| MAPPO/CTDE 协同路由 | 是否比单纯局部决策更均衡 | 去掉全局奖励/普通 MAPPO 消融，比较 Jain 负载公平性指数、吞吐量、投递成功率 |

队列感知有效时，不应只看到平均时延下降，还应看到最大队列长度和 95% 分位时延下降；链路寿命约束有效时，不应只看到丢包率下降，还应看到链路断开导致的重路由次数减少。

## 6. 实验验证思路

实验从两个角度验证：整体性能和模块贡献。

### 6.1 对比算法

可以考虑和以下方法对比：

- Dijkstra 最短路径；
- 虚拟拓扑路由；
- Q-routing 或 QRLSN；
- 普通 MAPPO；
- 本文完整方法。

各对比算法的作用如下：

| 对比方法 | 为什么要比 | 预期它在哪些场景会弱 |
|---|---|---|
| Dijkstra | 作为传统最短路径基线 | 高负载时可能把流量推向同一批短路径，队列变长 |
| 虚拟拓扑路由 | 代表基于周期性拓扑预计算的方法 | 突发流量或链路状态快速变化时适应慢 |
| Q-routing/QRLSN | 代表轻量级强化学习路由 | 状态表达能力有限，多智能体协同不足 |
| 普通 MAPPO | 验证多智能体学习本身的作用 | 如果不加队列/链路寿命/动作掩码，可能仍会选到拥塞或快断链路 |
| 本文完整方法 | 验证综合设计 | 目标是在高负载和链路频繁变化场景下更稳 |

这样可以区分“学习方法本身的作用”和“队列/链路寿命/动作掩码这些机制的作用”。

### 6.2 实验场景

计划设置几类典型场景：

1. **低负载场景**
   验证本文方法在轻负载下不会明显劣于最短路径。

2. **高负载热点场景**
   验证队列感知和负载感知是否能缓解拥塞。

3. **链路频繁断开场景**
   验证链路剩余可用时间约束是否能降低丢包和重路由。

4. **节点或链路故障场景**
   验证算法在异常情况下的鲁棒性。

场景需要给出可执行设置，初步如下：

| 场景 | 具体设置 | 主要想验证什么 |
|---|---|---|
| 低负载 | 包到达率低，队列基本不积压 | 验证方法不会因为绕路导致明显变差 |
| 中等负载 | 随机源目的对，链路利用率中等 | 验证综合性能 |
| 热点高负载 | 多个源同时向少数地面站/区域发送业务 | 验证队列感知和负载均衡 |
| 链路频繁断开 | 缩短部分 ISL 可用窗口或模拟跨轨链路断开 | 验证链路寿命约束 |
| 故障场景 | 随机提高部分链路中断概率或直接关闭链路 | 验证可靠性和恢复能力 |

高负载热点场景和链路频繁断开场景最关键，因为这两个场景最能体现本文和普通最短路/普通 MAPPO 的差别。

可执行的初始设置如下，数值只是为了把实验分成低负载、中负载、高负载几档，不作为文献结论：

| 参数 | 初始设想 | 作用 |
|---|---|---|
| 星座规模 | 先从 100-500 颗卫星仿真，再扩展 | 先保证实验能跑，再看规模泛化 |
| 每颗卫星候选链路 | 初始仿真可先用约 4 条 ISL，后续扩展到 `A_i(t)` 时变候选集合 | 先保证实验可控，再扩展到地面站链路、机会链路或分层路由下一跳 |
| 低负载 | 平均链路利用率取较低一档 | 队列不明显，主要看是否绕路 |
| 中等负载 | 平均链路利用率取中间一档 | 观察综合性能 |
| 高负载 | 平均链路利用率取较高一档 | 队列和拥塞开始明显 |
| 热点流量 | 让较大比例流量指向少数目的区域/地面站，并做比例扫描 | 模拟热点服务区 |
| 快断链路 | 设置部分链路 `T_rem < T_safe` 或频繁变化 | 验证链路寿命过滤 |
| 故障链路 | 随机选一部分链路提高中断概率，并改变故障比例 | 验证可靠性 |

这些数值只是为了让实验先跑起来，后续根据仿真规模、链路容量和对比算法结果调整。

### 6.3 评价指标

主要指标包括：

- 平均端到端时延；
- 吞吐量；
- 投递成功率；
- 丢包率；
- 平均队列长度；
- Jain 负载公平性指数；
- 路由切换次数；
- 控制消息开销。

后文 8.7 给出评价时怎么看这些指标。这里不提前把某个阈值写死，主要看完整方法相对基线是否同时改善时延、丢包、队列和稳定性；如果需要给工程约束，也放到敏感性实验里说明。

### 6.4 消融实验

消融实验包括：

- 去掉队列感知；
- 去掉链路负载项；
- 去掉链路剩余可用时间；
- 去掉动作掩码；
- 去掉全局协同奖励。

通过这些实验观察端到端时延、丢包率、队列长度和路由稳定性是否变差。

【待验证】消融实验重点观察：

| 去掉的模块 | 预期变差的指标 | 说明 |
|---|---|---|
| 去掉队列感知 | 平均队列长度、最大队列长度、端到端时延 | 说明队列项是否真的缓解拥塞 |
| 去掉链路负载 | Jain 负载公平性指数、最大链路利用率 | 说明负载项是否真的让流量更均衡 |
| 去掉链路剩余可用时间 | 重路由次数、丢包率、链路断开导致的失败次数 | 说明链路寿命约束是否有效 |
| 去掉动作掩码 | 无效动作比例、训练收敛速度、丢包率 | 说明过滤机制是否减少无效探索 |
| 去掉全局协同奖励 | 全网时延、Jain 负载公平性指数、吞吐量 | 说明全局目标是否改善多智能体协作 |

这样可以用消融结果回答每个因素对全局性能的影响大小。

## 8. 研究思维画布九步法梳理

这一节按“研究思维画布九步法”把整个课题重新串起来。它不是单纯重复前面的建模，而是说明：这个题目为什么值得做、具体卡在哪里、方案解决了哪一段问题、最后怎么证明。

### 8.1 用户场景

应用场景是高动态低轨卫星网络中的星间路由。研究对象不是单条链路，也不是单颗卫星，而是由大量 LEO 卫星组成的时变网络。

具体可以理解成下面这个转发过程：

```text
源地面站/用户
    ↓
接入卫星
    ↓
多跳星间链路转发
    ↓
目标区域附近卫星
    ↓
目的地面站/用户
```

在这个过程中，每颗中继卫星只负责一个很小的动作：给当前数据包选下一跳。但这个小动作会影响后面的队列、链路负载、丢包和重路由。

这个场景的特殊性在于：

| 特点 | 对路由的影响 |
|---|---|
| 卫星高速运动 | 星间链路和星地可见关系不断变化 |
| 星座规模大 | 全网状态收集和分发成本变高 |
| 流量不均匀 | 热点区域附近卫星容易排队 |
| 链路存在中断风险 | 当前可用链路可能很快不可用 |
| 星上资源有限 | 路由决策不能太复杂，也不能依赖大量控制消息 |

所以这个课题面向的不是“静态网络中求一条最短路”，而是“动态、拥塞、局部信息受限条件下的逐跳路由决策”。

### 8.2 研究需求

核心需求可以拆成一句话：

> 在低控制开销和低计算开销的前提下，让每颗卫星在线选出一个低时延、低拥塞、低失效风险的下一跳。

这里面有三个矛盾。

第一个矛盾是“全局最优目标”和“局部可观测信息”之间的矛盾。端到端时延、全网拥塞和负载均衡都是全局指标，但星上执行时每颗卫星只能看到自己和邻居。

第二个矛盾是“快速决策”和“状态新鲜度”之间的矛盾。更新越频繁，状态越新，但控制开销越大；更新越少，开销低，但卫星拿到的信息可能已经过期。

第三个矛盾是“短路径”和“稳定低拥塞路径”之间的矛盾。传播距离短不一定代表端到端时延低，因为下一跳可能队列很长，或者链路快断。

控制开销具体包括三类：

| 开销类型 | 来源 | 对本文的影响 |
|---|---|---|
| 状态收集开销 | 收集全网链路、队列、负载状态 | 星座越大越难实时收集 |
| 状态分发开销 | 将全网状态传播给所有节点 | 可能占用星间链路资源 |
| 重计算开销 | 拓扑或流量变化后重新计算路径 | 高动态场景下频繁触发 |
| 计算开销 | 星上执行动作过滤和 Actor 推理 | 直接影响每包/每流决策是否能实时完成 |

【估算】按前面的粗略估算，万颗级星座一次全网链路状态快照就可能是数百 KB 级，如果分发给所有卫星，总量就是 GB 级；如果几秒一次更新，控制流量会很可观。因此本文更倾向于局部邻居交换，而不是全网实时状态同步。

计算开销也需要单独回答。本文后续不只统计控制消息比例，还要统计：

```text
单次动作过滤时间
单次 Actor 推理时间
模型参数量
近似 FLOPs
```

如果 Actor 推理时间接近或超过单包调度时间，说明该方法即使控制开销低，也不适合直接星上部署。

对应到本文，就是：

```text
不用全网实时状态
    ↓
只依赖本地和邻居状态
    ↓
仍然尽量接近全局目标：低时延、少拥塞、少丢包、低开销
```

### 8.3 核心卖点

核心卖点不是“用了强化学习”，而是把 LEO 路由里几个容易被忽略的因素真正放进决策里。

1. **基于队列和负载状态进行下一跳决策**
   不只看传播时延，也看下一跳是否拥塞。如果邻居 `j` 的队列已经很长，即使 `i -> j` 链路传播时延低，也可能导致端到端时延升高。

2. **基于时变候选集合进行动作过滤**
   不把动作空间固定成四个邻居，而是对当前时刻所有候选链路 `A_i(t)` 做过滤，包括普通 ISL、地面站链路、机会链路或分层路由允许的下一跳。这个设计主要针对 LEO 链路高动态问题。

3. **基于 MAPPO/CTDE 实现集中训练、分布式执行**
   训练时利用全局状态学习协同策略，执行时每颗卫星只用局部观测独立决策。这样既能利用全局奖励训练协作行为，又不要求星上执行时依赖地面中心实时控制。

三个卖点对应的“局部动作 -> 全局结果”关系如下。下面这些影响属于【待验证】，后续用实验结果填实：

| 卖点 | 局部决策改变了什么 | 主要影响的全局指标 | 预期结果 |
|---|---|---|---|
| 队列和负载感知 | 避免把包继续推向拥塞邻居 | 最大队列长度、Jain 负载公平性指数、95% 分位时延 | 热点拥塞减轻，尾部时延下降，负载公平性提高 |
| 时变动作过滤 | 避免选择不可用、快断或高风险链路 | 无效动作比例、丢包率、重路由次数 | 链路失效导致的失败减少 |
| MAPPO/CTDE 协同训练 | 让多个卫星的局部策略朝同一全局奖励优化 | 投递成功率、吞吐量、Jain 负载公平性指数 | 减少局部最优导致的全网拥塞 |

这里的关键点是：每个卖点都要能落到实验上，否则容易变成单纯把因素堆进模型。

### 8.4 解决方案

整体解决思路分成四步。

**第一步：网络建模。**
把 LEO 网络建模为时变图 `G(t) = (V, E(t), X(t))`。`E(t)` 表示随时间变化的链路，`X(t)` 表示队列、带宽、负载、可靠性、链路剩余时间等状态。

**第二步：智能体建模。**
每颗卫星是一个智能体。智能体不计算完整端到端路径，只在每次转发时选择下一跳。

**第三步：动作过滤。**
在 Actor 输出动作前，先过滤不可用链路、低带宽链路、高队列邻居、低可靠链路、快断链路和可能形成环路的动作。

**第四步：MAPPO/CTDE 训练与执行。**
训练阶段使用集中式 Critic 和全局奖励，执行阶段只部署 Actor 策略网络。每颗卫星根据自身和邻居状态独立选择下一跳。

执行时可以写成下面的步骤：

```text
卫星 i 收到数据包
    ↓
读取自身队列 q_i
    ↓
通过 Hello 包获取邻居队列 q_j、链路负载 rho_ij、可靠性、T_rem_ij
    ↓
动作掩码过滤不可用/高风险下一跳
    ↓
Actor 在候选邻居中选择 j
    ↓
转发数据包，并记录时延、队列、丢包、切换等信息
```

### 8.5 重要意义

方案有效时，意义不只是“路由性能更好”，而是能说明分布式学习型路由在 LEO 场景里有实际价值。

| 意义 | 具体体现 | 需要看到的实验现象 |
|---|---|---|
| 降低对地面中心实时控制的依赖 | 星上只用局部观测和邻居交换完成下一跳选择 | 控制开销低于全局链路状态类方法 |
| 缓解热点拥塞 | 不把流量持续压到同一批短路径 | 最大队列长度下降，Jain 负载公平性指数上升 |
| 提升链路动态环境下的稳定性 | 提前规避快断链路 | 链路断开导致的丢包和重路由次数下降 |
| 保持业务投递能力 | 高负载下仍能维持较高投递成功率 | 投递成功率和吞吐量不低于对比算法 |

其中最关键的是不要只看平均时延。如果平均时延下降，但丢包率上升、路由切换频繁，说明方法并不稳定。因此至少要同时观察：

```text
平均时延 + 95% 分位时延 + 丢包率 + 最大队列 + 路由切换次数
```

### 8.6 代价

这个方案也有代价，主要集中在训练、邻居信息交换和星上推理三个方面。

| 代价 | 为什么会出现 | 怎么量化 | 如果太大怎么办 |
|---|---|---|---|
| 集中训练成本 | MAPPO/CTDE 需要大量仿真交互 | 训练轮数、训练总时长、收敛曲线 | 降低状态维度，先小规模训练再泛化 |
| 仿真到真实偏差 | 仿真链路和真实星上环境不完全一致 | 不同星座规模、不同流量模式下测试 | 做泛化实验和参数扰动实验 |
| Hello 控制开销 | 卫星需要交换邻居队列和链路状态 | `control_bytes / data_bytes` | 拉长 Hello 周期，压缩状态字段 |
| 星上计算开销 | Actor 推理需要星上算力 | 单次推理时间、模型参数量 | 模型压缩、减少输入维度 |
| 奖励权重调参 | 多目标奖励容易互相冲突 | 权重敏感性实验 | 固定主目标，逐项调参 |

这里不直接说“超过多少就一定不行”。更稳的写法是把 Hello 周期和状态字段长度做成敏感性实验：如果控制开销随 Hello 频率上升很快，或者 Actor 推理时间接近单包调度时间，就需要调整 Hello 周期、压缩状态字段或压缩 Actor 模型。

### 8.7 核心技术指标

核心指标分成五组，每一组回答一个问题。

| 指标组 | 具体指标 | 回答的问题 |
|---|---|---|
| 时延性能 | 平均端到端时延、95% 分位时延 | 方法是不是更快，尾部拥塞有没有改善 |
| 吞吐和可靠性 | 吞吐量、投递成功率、丢包率 | 包能不能稳定送到 |
| 拥塞情况 | 平均队列长度、最大队列长度、最大链路利用率、Jain 负载公平性指数 | 是否缓解局部热点拥塞 |
| 稳定性 | 路由切换次数、重路由次数、链路断开后的恢复时间 | 是否减少路径震荡和链路失效影响 |
| 开销 | Hello 包控制开销、单次动作过滤时间、Actor 推理时间、模型参数量、近似 FLOPs | 是否真的适合星上分布式执行 |

关键不是单独看平均时延，而是同时看：

```text
平均端到端时延 + 最大队列长度 + 丢包率 + 路由切换次数
```

因为如果只看平均时延，可能看不出拥塞和链路失效问题；如果只看吞吐，也可能忽略局部热点队列。

评价时先不写死绝对阈值，而是看相对基线和消融版本的变化：

| 指标 | 比较好的状态 | 变差时说明什么 | 对应设计 |
|---|---|---|---|
| 平均端到端时延 | 相比 Dijkstra/Q-routing 下降，或不高于最短路太多 | 绕路过多或队列惩罚设置不合理 | 队列感知、负载感知 |
| 95% 分位时延 | 明显下降 | 尾部拥塞没有缓解 | 队列感知、全局奖励 |
| 投递成功率 | 相比基线提高，或者在高负载/故障场景下下降更少 | 丢包、链路失效或环路严重 | 动作掩码、可靠性过滤 |
| 丢包率 | 相比基线下降，尤其是快断链路和热点场景 | 拥塞或链路中断明显 | 队列惩罚、链路寿命约束 |
| 平均队列长度 | 随负载增加时增长更慢 | 全网拥塞压力较大 | 队列奖励项 |
| 最大队列长度 | 热点节点的峰值队列下降 | 出现持续热点节点 | 负载均衡奖励 |
| Jain 负载公平性指数 | 越接近 1 越好，完整方法应高于去掉负载项的版本 | 流量集中在少数链路 | 链路负载项、全局奖励 |
| 路由切换率 | 链路快断场景下不应频繁震荡 | 下一跳选择不稳定 | 切换惩罚、链路寿命过滤 |
| 控制开销比例 | 相比全局链路状态类方法更低 | 状态交换过多 | Hello 周期和状态字段压缩 |

这些不是固定结论，而是实验时的观察方向。真正写论文时，应优先报告“相对基线改善多少”和“参数变化后是否稳定”，而不是只给一个没有来源的绝对阈值。

这些判断标准还需要做阈值敏感性分析。例如把队列过滤阈值、可靠性过滤阈值、`T_safe` 和 Hello 周期分别调高/调低，观察时延、丢包、重路由和控制开销是否稳定。

### 8.8 进一步改进空间

这一部分写成“哪些地方可能把实验结论推翻”。也就是说，不只是说以后能继续做，而是提前说明哪些假设需要被检查。

| 不完善点 | 为什么会影响结论 | 可能影响哪些指标 | 后续怎么检查 |
|---|---|---|---|
| 泛化不足 | MAPPO 在一个星座规模上学到的策略，可能记住的是局部拓扑规律，而不是一般路由规律 | 换到更大星座后，平均时延、投递成功率、最大队列可能变差 | 先在 100 颗训练，再在 300、500 颗测试；再反过来做大规模训练、小规模测试 |
| 链路寿命估计不准 | `T_rem` 直接决定一条链路会不会被 mask 掉，估计错了会改变动作集合 | 误杀好链路会增加绕路和平均时延；漏掉快断链路会增加丢包和重路由 | 在真实 `T_rem` 上人为加入 5%、10%、20% 误差，看丢包率、重路由次数和平均跳数变化 |
| 奖励权重敏感 | 队列、时延、丢包之间不是同一个目标，权重一变，策略可能偏向不同方向 | `w_Q` 太大可能绕路，`w_L` 太大可能挤短路径，`w_P` 太大可能过度保守 | 固定场景，只改一组权重，画平均时延、95% 分位时延、最大队列、丢包率的变化 |
| 变量过多导致状态空间膨胀 | 如果把所有可观测量都放进 Actor，训练更慢，Hello 包更大，星上推理也更重 | 收敛速度下降、Actor 推理时间增加、控制开销增加，甚至性能不升反降 | 做变量消融和逐步加变量实验：先用核心变量，再分别加入可靠性、`T_rem`、`age_j`，看收益是否大于开销 |
| 邻居信息过期 | Hello 包里的 `q_j` 和 `rho_ij` 有时间差，高动态和高负载下旧信息可能误导下一跳选择 | Hello 周期过长时，最大队列、丢包率、路由切换次数可能上升 | 比较 0.5 s、1 s、2 s、5 s 等 Hello 周期，同时记录控制开销比例 |
| 星上算力有限 | Actor 最后要放到卫星上，不能只看仿真性能 | 推理时间过长会影响每包决策；模型过大则不适合部署 | 统计单次动作过滤时间、Actor 推理时间、参数量和 FLOPs；与 SHORT 中首包处理 0.048-0.058 ms、FIB 查找 0.006-0.008 ms 这种量级做参考对照 |
| 业务类型单一 | 低时延业务、普通下载业务和突发热点业务的目标不同，一个奖励函数可能不能同时照顾 | 低时延业务可能被绕路，普通业务可能占用关键链路 | 后续把业务分成 delay-sensitive 和 throughput-sensitive 两类，再比较不同业务的时延和吞吐 |
| 对比算法不够强 | 如果只和 Dijkstra 或普通 Q-routing 比，基线说服力不够 | 方法优势可能被高估 | 至少加入普通 MAPPO、QRLSN/GraphPR 类分布式学习路由、虚拟拓扑路由；如果实现不了，需要说明只做思想对比 |

最优先要补的是四件事：第一，变量取舍是否合理；第二，`T_rem` 估计误差；第三，Hello 周期导致的状态过期；第四，跨规模泛化。因为这些如果表现不好，本文就会从“分布式动态路由”退化成“只在一个理想仿真里有效的路由策略”。

这一节后面可以按 IEEE 论文的实验结构继续细化成四组实验：

| 实验组 | 目的 | 关键结果应该怎么解释 |
|---|---|---|
| 变量消融 | 证明不是变量堆砌 | 每加入一个变量，都要说明它主要改善哪个指标，以及带来多少推理/控制开销 |
| 阈值敏感性 | 证明 `T_safe`、可靠性阈值、队列阈值不是主观设定 | 参数变化后趋势仍稳定，才说明方法不依赖单点调参 |
| 跨场景泛化 | 证明不是只适配一个星座或一种流量 | 在不同规模、不同热点比例、不同链路故障率下看性能是否保持 |
| 开销对照 | 证明分布式执行真的可部署 | 统计 Hello 开销、动作过滤时间、Actor 推理时间，并和全局路由更新开销做数量级对比 |

### 8.9 回到原点

短期内更适合先在仿真平台中验证，比如自建 LEO 网络仿真环境，或者结合 ns-3、STK 等工具生成拓扑和链路变化。等仿真结果稳定后，再考虑原型平台或更真实的系统验证。

阶段目标不是直接部署到真实卫星，而是先证明三件事：

1. **局部观测够不够用**：只用自身和邻居状态，能不能接近或超过全局路由类方法的性能；
2. **队列和链路寿命是否真的有用**：去掉这两个因素后，队列、丢包、重路由是否明显变差；
3. **分布式执行是否低开销**：Hello 包和 Actor 推理开销是否低于全局状态维护和频繁重计算。

如果这三点成立，这个方向就可以继续往论文实验推进；如果不成立，就需要回到状态设计、奖励函数或 Hello 交换机制重新调整。

## 9. 目前总结

整体思路：

> 先把 LEO 卫星网络建模为时变图，再用 Dec-POMDP 表达多卫星协作和局部可观测；在此基础上参考 RTMDP，把观测过期、动作延迟和估计误差放进实时增强观测。每颗卫星只利用自身状态、一跳邻居状态和相邻链路状态进行下一跳选择。为了避免拥塞和链路失效，在状态和奖励中加入队列长度、链路负载、链路可靠性和链路剩余可用时间，并通过动作掩码过滤明显不可用或高风险链路。方法上采用 MAPPO/CTDE 思路，实现训练阶段全局协同、执行阶段分布式自治。实验上通过对比实验和消融实验验证该方法在时延、丢包率、负载均衡和路由稳定性方面的效果。

关键逻辑链：

```text
LEO 高动态 + 万颗级规模
        ↓
全局状态维护和集中式最短路开销大
        ↓
只能依赖局部观测和邻居交换
        ↓
局部最短路又容易忽略队列、负载、链路寿命
        ↓
加入队列感知、链路寿命约束和动作掩码
        ↓
用 MAPPO/CTDE 在训练阶段学全局协同，执行阶段保持分布式
        ↓
通过时延、队列、丢包、重路由、控制开销验证是否有效
```

后续需要进一步细化的地方包括：

- 状态空间如何进一步压缩，哪些变量应进入 Actor，哪些只进入掩码、Critic 或评价指标；
- 链路剩余可用时间如何计算；
- 奖励函数各项权重如何设置；
- 仿真星座和业务流量如何生成；
- 与已有 LEO 路由算法的公平对比方式。

下一步重点不是继续加概念，而是定下四个东西：

1. **仿真场景参数**：星座规模、链路容量、业务到达率、热点流量比例；
2. **变量取舍表**：核心 Actor 输入、动作掩码变量、Critic 全局变量、评价指标分别是什么；
3. **奖励函数参数**：`D_target`、`Q_max`、`T_safe` 和各项权重；
4. **消融实验表**：每个模块或变量去掉后观察哪些指标变化。

这样后续讨论“如何证明”时，可以直接对应到具体实验和指标。
