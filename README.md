# 低轨卫星网络分布式动态路由研究思路

这是我目前对研究方向的一个梳理。我想把题目先收敛到：

**基于 PS-MAPPO 的队列感知与链路寿命约束低轨卫星网络分布式动态路由研究**

我现在的理解是，这个方向不能只写成“用强化学习做卫星路由”，这样太宽，也不好证明创新点。所以我想把问题聚焦到一个更具体的点：

**在低轨卫星高速运动、拓扑频繁变化的情况下，每颗卫星只依赖自身和一跳邻居信息，如何选择一个低拥塞、低时延、并且不容易失效的下一跳。**

---

## 1. 我想解决的具体问题

低轨卫星网络和普通地面网络不太一样。我现在读文献之后，觉得这个场景里最麻烦的不是“能不能算一条最短路”，而是**这个最短路在高动态和高负载情况下还靠不靠谱**。

先把场景说具体一点：

- **规模很大**：以 Starlink 为例，Space.com 在 2026 年 6 月上旬的报道中提到，Starlink 活跃卫星数量已经超过 10,500 颗，后续报道还提到已超过 10,580 颗。这个数量级下，全网状态收集和全局路径计算本身就会很重。  
  参考：https://www.space.com/spacex-starlink-satellites.html

- **移动很快**：低轨卫星维持轨道的平均速度大约是 7.8 km/s，也就是约 28,000 km/h。这个速度下，星间链路和星地可见关系都会随时间不断变化。  
  参考：https://en.wikipedia.org/wiki/Low_Earth_orbit

- **集中式路由开销确实是问题**：Page 等人在 *Distributed Probabilistic Congestion Control in LEO Satellite Networks* 中明确指出，在密集 LEO 星座中使用集中式最小时延路由会带来显著的信令和计算开销，所以他们才转向基于邻居最新流量信息的分布式拥塞控制。  
  参考：https://arxiv.org/abs/2209.08565

- **队列和实时流量变化不能忽略**：2026 年的 *Queue-Aware and Resilient Routing in LEO Satellite Networks Using Multi-Agent Reinforcement Learning* 里提到，传统路由通常假设静态链路指标，不能很好处理 queue backlogs 和实时系统变化；同一篇文章还提到，在 5 s 重计算间隔下，其方法的开销大约只有 Dijkstra 的 50%。  
  参考：https://arxiv.org/abs/2605.04448

- **热点区域会放大拥塞问题**：Cao 等人在 *Computing-Aware Routing for LEO Satellite Networks* 中提到，热点区域会让可见 LEO 卫星的有限计算/传输资源更加紧张；他们的 computing-aware routing 相比把原始数据直接下传到地面处理，整体时延最高下降 78.31%。虽然这篇重点是计算感知路由，但它说明热点区域和资源受限确实会明显影响路由性能。  
  参考：https://arxiv.org/abs/2211.08820

所以我目前不是想泛泛地做“LEO 路由优化”，而是想做一个更具体的问题：

> 在局部信息受限的条件下，如何让每颗卫星根据邻居队列、链路负载和链路剩余可用时间，分布式地选择下一跳，从而降低端到端时延、丢包率和路由震荡。

这里面我重点关注三个痛点：

1. **局部信息下的快速路由决策**  
   万颗级星座里，如果每次拓扑或流量变化都依赖全局状态收集，再集中计算最短路径，信令和计算开销会很高。已有文献也明确指出密集 LEO 星座中集中式最小时延路由存在显著开销问题。所以我想让每颗卫星只依赖自身和一跳邻居信息做下一跳选择。

2. **队列积压和链路拥塞**  
   最短路径不一定是实际最快路径。如果下一跳队列很长，或者链路已经很忙，即使传播时延低，也可能造成更大的端到端时延。Queue-aware LEO routing 的文献也说明，传统静态链路指标没有充分考虑 queue backlogs 和实时流量变化，这会影响路由效果。

3. **链路即将断开的问题**  
   LEO 星间链路具有明显的时间动态性。当前可用的链路可能很快失效，如果路由选择了这种链路，可能导致重路由、丢包和额外时延。所以我想把链路剩余可用时间作为动作过滤和奖励设计中的一个重要因素。

简单说，我现在想抓住的核心不是“最短路怎么求”，而是：

> **在局部信息、实时拥塞和链路高动态同时存在的情况下，怎么让卫星更稳地选下一跳。**

## 2. 整体建模思路

我打算先从网络本身开始建模，然后再引出多智能体强化学习。

### 2.1 时变图建模

因为 LEO 网络拓扑随时间变化，所以可以先把网络建成一个时变图：

```text
G(t) = (V, E(t), X(t))
```

其中：

- `V` 表示卫星节点集合；
- `E(t)` 表示 t 时刻可用的星间链路；
- `X(t)` 表示节点和链路状态，比如队列、带宽、负载、可靠性、链路剩余可用时间等。

这样建模的原因是：LEO 网络最核心的特点就是链路会随着时间变化，不能简单当作静态图处理。

这里我想把“控制开销大”说得更具体一点。假设一个万颗级星座里有 `N = 10,000` 颗卫星，每颗卫星平均维护 4 条星间链路，那么无向链路数大约是：

```text
|E| ≈ N × 4 / 2 = 20,000
```

如果每条链路状态至少包含：

```text
链路两端 ID、传播时延、剩余带宽、负载率、可靠性、链路剩余时间、时间戳
```

即使非常粗略地按每条链路 32 B 计算，一次全网链路状态快照也大约是：

```text
20,000 × 32 B ≈ 640 KB
```

这个 640 KB 只是原始链路状态，不包括协议头、转发冗余、可靠广播开销和多跳传播。如果要求每颗卫星都拿到全网状态，那么一次全网同步的总分发量可以达到：

```text
640 KB × 10,000 ≈ 6.4 GB
```

如果每 5 s 更新一次，纯状态分发的数量级就是：

```text
6.4 GB / 5 s ≈ 1.28 GB/s
```

这个估算不是某篇论文的实验值，而是为了说明一个数量级：**在万颗级星座里，全网链路状态同步很容易变成很重的控制负担**。这也能解释为什么很多文献会转向分布式、局部邻居交换或学习型路由。2605.04448 那篇 queue-aware MARL 文章也直接指出，在 5 s 重计算间隔下，它的方法开销大约只有 Dijkstra 的 50%，说明全局最短路类方法的重计算和状态维护确实会成为瓶颈。

### 2.2 智能体建模

每颗卫星看作一个智能体。卫星在转发数据包时，只需要决定当前数据包的下一跳，而不是一次性计算完整路径。

这样做的好处是：

- 更符合分布式执行；
- 可以根据实时队列和链路状态调整；
- 不需要频繁维护全局端到端路径。

这里其实有一个取舍：逐跳路由不保证每一步都得到全局最短路径，但它更适合高动态场景。因为如果提前算完整路径，路径中间某条链路断了，整条路径就要重算；逐跳路由虽然可能局部绕一下，但每一步都能根据当前状态重新判断。

所以我后面要验证的不是“每条路径是不是最短”，而是：

```text
逐跳分布式决策能不能在总体上降低丢包、队列积压和链路失效带来的重路由。
```

如果实验结果显示平均跳数略有增加，但 95% 分位时延、丢包率和重路由次数下降，那我认为这个取舍是合理的。

### 2.3 局部观测设计

每颗卫星能够观测的信息主要包括：

- 自身队列长度；
- 邻居卫星队列长度；
- 相邻链路传播时延；
- 链路剩余带宽；
- 链路负载率；
- 链路可靠性或中断概率；
- 链路剩余可用时间；
- 当前数据包的目的节点或目的区域。

我加入这些状态的原因是：

- 队列长度用于判断下一跳是否拥塞；
- 链路负载用于判断链路是否繁忙；
- 传播时延用于判断基础传输距离；
- 剩余带宽用于判断链路传输能力；
- 可靠性用于判断传输失败风险；
- 链路剩余可用时间用于判断链路是否快要断开。

更具体一点，我现在想把局部观测拆成三组：

```text
自身状态：q_i, buffer_i, service_rate_i
邻居状态：q_j, load_j, available_j
链路状态：delay_ij, bandwidth_ij, rho_ij, reliability_ij, T_rem_ij
```

其中我最想强调的是 `q_j` 和 `T_rem_ij`：

- `q_j` 解决的是“下一跳现在堵不堵”；
- `T_rem_ij` 解决的是“这条链路会不会很快断”。

2605.04448 那篇文章里把单跳时延写成传播时延、传输时延和排队时延的组合，说明队列项不是随便加的，而是会直接进入时延。也就是说，如果只看传播距离，实际可能选到一条“距离短但队列很长”的路径。

### 2.4 动作设计

动作就是选择下一跳：

```text
a_i(t) = 选择某个邻居卫星作为下一跳
```

如果当前卫星有多个邻居，就从这些邻居里面选择一个。

### 2.5 动作过滤

在真正选择下一跳之前，我想先做一层动作过滤。也就是先把明显不适合的链路排除掉，比如：

- 链路当前不可用；
- 剩余带宽不足；
- 下一跳队列过长；
- 链路可靠性过低；
- 链路剩余可用时间太短；
- 可能形成路由环路。

这样做的原因是，强化学习不应该完全从零开始学习这些明显的硬约束。先过滤掉不可行动作，可以减少无效探索，也能提高路由稳定性。

动作过滤可以写成下面这种规则：

```text
候选下一跳 j 必须同时满足：
1. link_ij 当前可用
2. bandwidth_ij > B_min
3. q_j < Q_max
4. reliability_ij > R_min
5. T_rem_ij > T_safe
6. j 不在当前数据包已访问节点集合中
```

其中 `T_safe` 不能只随便设一个常数。我现在的想法是让它至少大于一次转发可能需要的时间：

```text
T_safe > T_queue + T_transmission + T_propagation + T_margin
```

这样做的含义是：如果这条链路剩余可用时间连当前数据包安全通过都不够，那就算它现在可用，也不应该选。

这个部分可以作为我和已有普通 MAPPO 路由的区别：不是直接让策略网络在所有邻居里选，而是先用网络约束做 action mask，再让 PS-MAPPO 在可行动作里学选择。

## 3. 方法选择：PS-MAPPO

我目前打算采用 **PS-MAPPO** 作为具体方法。这里 PS-MAPPO 指：

**Parameter Sharing Multi-Agent Proximal Policy Optimization**

也就是：

```text
参数共享 Actor + 集中式 Critic + 分布式执行 + 动作掩码
```

选择这个方法的原因是：

1. **这是一个多智能体问题**  
   每颗卫星的转发决策都会影响其他卫星的队列和链路负载，所以不能只看单个节点的局部最优。

2. **适合 CTDE 思路**  
   训练阶段可以在地面仿真平台获取全局状态，所以可以使用集中式 Critic；但真正执行时，每颗卫星只能用本地观测，所以只部署 Actor。

3. **卫星节点比较同构**  
   大多数卫星的转发功能类似，因此可以让所有卫星共享同一个 Actor 网络，减少模型参数量和星上存储开销。

4. **动作是离散的**  
   路由动作就是从邻居中选择一个下一跳，比较适合 MAPPO 这种离散动作策略方法。

5. **方便加入动作掩码**  
   LEO 网络中邻居集合会变化，部分链路也会不可用，所以动作掩码比较重要。

训练阶段：

- Critic 可以看到全局网络状态；
- 用全局奖励引导多颗卫星协同；
- 目标是降低全网时延、队列积压、丢包率和路由切换。

执行阶段：

- 每颗卫星只使用共享 Actor；
- 输入是局部观测和邻居状态；
- 输出是下一跳选择；
- 不依赖地面中心实时控制。

这里还需要说清楚：为什么不是直接用其他方法。

| 方法 | 主要问题 | 为什么不作为主方案 |
|---|---|---|
| Dijkstra | 依赖较新的全局链路状态，主要优化最短路径 | 高动态和高负载下容易忽略队列、链路寿命和状态更新开销 |
| Q-routing | 分布式、轻量 | 表格型方法表达能力有限，难同时处理队列、负载、可靠性、链路寿命等连续状态 |
| DQN | 可以处理复杂状态 | 更偏单智能体，难直接表达多颗卫星之间的协作关系 |
| QMIX | 适合合作 MARL | 动作空间通常要求更规整，而 LEO 邻居集合会随时间变化，动作掩码处理不如 MAPPO 直接 |
| 普通 MAPPO | 能做多智能体协作 | 如果不加参数共享和动作掩码，星上部署成本和不可用链路处理都不够清楚 |

所以我选择 PS-MAPPO 的具体理由是：

```text
PS-MAPPO = MAPPO 的协同训练能力 + 参数共享降低部署成本 + 动作掩码处理动态链路约束
```

这个选择也需要实验验证。比如比较：

```text
普通 MAPPO vs PS-MAPPO + 动作掩码
```

如果加入动作掩码后，无效动作比例下降、训练收敛更快、丢包率下降，那就说明这个设计不是随便加的。

## 4. 奖励函数设计思路

奖励函数这里不能只写“越高越好/越低越好”，不然确实像在堆指标。我现在的想法是：先把每个指标变成 `[0, 1]` 范围内的量，再决定它是正奖励还是惩罚，最后再通过权重体现我更重视什么。

我准备把奖励分成两层：

```text
训练阶段：全局协同奖励 R_global
单跳选择：局部即时奖励 r_local
```

全局奖励解决“大家别只顾自己”的问题；局部奖励解决“这一跳到底选哪个邻居”的问题。

### 4.1 全局协同奖励

全局奖励我先设计成下面这种形式：

```text
R_global =
  + w_T * S_throughput
  + w_D * S_delivery
  - w_L * C_delay
  - w_Q * C_queue
  - w_B * C_load_balance
  - w_P * C_loss
  - w_S * C_switch
  - w_O * C_overhead
```

这里的 `S` 表示 score，越大越好；`C` 表示 cost，越小越好。

| 指标 | 怎么算 | 多高/多低算好 | 对奖励的影响 |
|---|---|---|---|
| `S_throughput` | `delivered_bits / offered_bits` | `>0.90` 较好，`<0.70` 较差 | 正奖励，越大越好 |
| `S_delivery` | `delivered_packets / generated_packets` | `>0.95` 较好，`<0.90` 较差 | 正奖励，越大越好 |
| `C_delay` | `min(avg_delay / D_target, 1)` | `<0.5` 较好，`=1` 表示达到或超过时延阈值 | 惩罚，越小越好 |
| `C_queue` | `0.5*avg_queue/Q_max + 0.5*max_queue/Q_max` | `<0.3` 健康，`>0.7` 拥塞严重 | 惩罚，越小越好 |
| `C_load_balance` | `std(rho)/(mean(rho)+epsilon)` | `<0.3` 较均衡，`>0.6` 热点明显 | 惩罚，越小越好 |
| `C_loss` | `dropped_packets / generated_packets` | `<0.01` 较好，`>0.05` 较差 | 强惩罚 |
| `C_switch` | `switch_count / forwarding_decision_count` | `<0.1` 稳定，`>0.3` 可能震荡 | 中等惩罚 |
| `C_overhead` | `control_bytes / data_bytes` | `<0.01` 很低，`>0.05` 明显 | 轻到中等惩罚 |

举个例子，如果某个训练阶段里：

```text
S_throughput = 0.85
S_delivery = 0.93
C_delay = 0.60
C_queue = 0.55
C_load_balance = 0.40
C_loss = 0.03
C_switch = 0.20
C_overhead = 0.02
```

那说明这时网络不是完全坏掉，但已经有一定拥塞和时延压力。这个奖励会鼓励模型继续提高吞吐和投递成功率，同时压低队列、时延和丢包。

权重我不想直接拍脑袋定死，但可以先按重要性排序：

```text
投递成功率/丢包  >  时延  >  队列/负载均衡  >  吞吐  >  路由切换  >  控制开销
```

初始权重可以先设成：

```text
w_D = 2.0   投递成功率
w_P = 2.0   丢包惩罚
w_L = 1.5   时延惩罚
w_Q = 1.0   队列惩罚
w_B = 1.0   负载不均衡惩罚
w_T = 1.0   吞吐奖励
w_S = 0.5   切换惩罚
w_O = 0.3   控制开销惩罚
```

这么设的原因是：如果包都送不到，吞吐或时延再好也没意义；所以投递成功率和丢包权重要更高。控制开销也重要，但它不能压过路由质量，否则模型可能为了少交换状态而选错路。

后面需要做权重敏感性实验，比如：

- 增大 `w_Q`，看队列是否下降，但路径是否绕远；
- 增大 `w_S`，看路由是否更稳定，但拥塞恢复是否变慢；
- 增大 `w_T`，看吞吐是否提高，但丢包是否上升；
- 增大 `w_O`，看控制开销是否下降，但状态过期是否让性能变差。

全局奖励的作用也要通过实验量化。我要比较：

```text
完整方法 vs 去掉全局协同奖励
```

如果去掉全局奖励后，平均时延变化不大，但最大队列长度、链路负载方差和丢包率上升，就说明全局奖励主要作用在削峰和负载均衡，而不是只优化平均值。

### 4.2 局部即时奖励

局部即时奖励主要负责某一跳怎么选。一个初步形式可以写成：

```text
r_local(i, j) =
  + eta_1 * I_deliver
  - eta_2 * delay_ij_norm
  - eta_3 * queue_j_norm
  - eta_4 * load_ij_norm
  - eta_5 * risk_ij
  - eta_6 * lifetime_penalty_ij
  - eta_7 * I_loop
  - eta_8 * I_switch
```

每一项对应的意思是：

| 项 | 怎么算 | 多高算危险 | 影响 |
|---|---|---|---|
| `delay_ij_norm` | 单跳传播+传输时延归一化 | `>0.7` 表示这条链路时延偏高 | 抑制过远链路 |
| `queue_j_norm` | `q_j / Q_max` | `>0.7` 表示下一跳可能拥塞 | 避免队列长的邻居 |
| `load_ij_norm` | 链路利用率 `rho_ij` | `>0.8` 表示链路接近满载 | 避免热点链路 |
| `risk_ij` | `1 - reliability_ij` 或中断概率 | `>0.1` 就需要惩罚，`>0.3` 应该直接过滤 | 降低丢包 |
| `lifetime_penalty_ij` | 和链路剩余时间有关 | 剩余时间越短越危险 | 避免快断链路 |
| `I_loop` | 形成环路为 1 | 一旦为 1 就强惩罚 | 避免循环转发 |
| `I_switch` | 相比上次切换下一跳为 1 | 频繁出现说明路由震荡 | 抑制震荡 |

链路剩余时间我想单独设计得更硬一点：

```text
if T_rem_ij < T_safe:
    直接 mask 掉
else:
    lifetime_penalty_ij = T_safe / T_rem_ij
```

比如：

```text
T_safe = 2 s
T_rem = 10 s  -> penalty = 0.2
T_rem = 3 s   -> penalty = 0.67
T_rem = 1 s   -> 直接不能选
```

这样就不是简单说“链路剩余时间越长越好”，而是明确说明：**剩余时间低于安全阈值直接过滤，高于阈值但越接近阈值惩罚越大。**

局部奖励的目标不是让每一跳都选最短，而是在下面四个因素之间折中：

```text
短路径 + 低队列 + 低负载 + 不快断
```

这也解释了为什么有时候算法可能不选传播时延最小的邻居：如果那个邻居队列很长或者链路快断，选它反而会让端到端结果更差。

## 5. 预期创新点

目前我认为可以提炼成三个创新点：

1. **队列感知的分布式路由决策**  
   在下一跳选择中加入邻居队列长度和链路负载，避免流量集中到局部最短路径。

2. **链路寿命约束的动作过滤机制**  
   在策略网络选择动作前，过滤即将断开或可靠性较低的链路，减少路径中断和重路由。

3. **基于 PS-MAPPO 的 CTDE 协同路由框架**  
   训练阶段利用全局状态学习协同策略，执行阶段每颗卫星只根据局部信息独立决策。

这三个创新点不能只写成“我提出了什么”，还要对应到“我怎么证明它有用”：

| 创新点 | 需要证明什么 | 对应实验 |
|---|---|---|
| 队列感知 | 加入队列后是否真的缓解拥塞 | 高负载热点场景，比较平均队列、最大队列、端到端时延 |
| 链路寿命约束 | 是否减少快断链路导致的失败 | 链路频繁断开场景，比较丢包率、重路由次数、路径中断次数 |
| PS-MAPPO 协同路由 | 是否比单纯局部决策更均衡 | 去掉全局奖励/普通 MAPPO 消融，比较负载方差、吞吐量、投递成功率 |

我现在希望每个创新点至少有一组对比实验支撑。比如队列感知如果有效，应该不只是平均时延下降，还应该看到最大队列长度下降；链路寿命约束如果有效，应该不只是丢包率下降，还应该看到由于链路断开导致的重路由次数减少。

## 6. 实验验证思路

实验部分我打算从两个角度验证：整体性能和模块贡献。

### 6.1 对比算法

可以考虑和以下方法对比：

- Dijkstra 最短路径；
- 虚拟拓扑路由；
- Q-routing 或 QRLSN；
- 普通 MAPPO；
- 本文完整方法。

每个对比算法的作用不一样：

| 对比方法 | 为什么要比 | 我预期它在哪些场景会弱 |
|---|---|---|
| Dijkstra | 作为传统最短路径基线 | 高负载时可能把流量推向同一批短路径，队列变长 |
| 虚拟拓扑路由 | 代表基于周期性拓扑预计算的方法 | 突发流量或链路状态快速变化时适应慢 |
| Q-routing/QRLSN | 代表轻量级强化学习路由 | 状态表达能力有限，多智能体协同不足 |
| 普通 MAPPO | 验证多智能体学习本身的作用 | 如果不加队列/链路寿命/动作掩码，可能仍会选到拥塞或快断链路 |
| 本文完整方法 | 验证综合设计 | 目标是在高负载和链路频繁变化场景下更稳 |

这样设计对比的目的，是把“学习方法的作用”和“队列/链路寿命/动作掩码这些机制的作用”分开看。

### 6.2 实验场景

计划设置几类典型场景：

1. **低负载场景**  
   验证本文方法在轻负载下不会明显劣于最短路径。

2. **高负载热点场景**  
   验证队列感知和负载感知是否能缓解拥塞。

3. **链路频繁断开场景**  
   验证链路剩余可用时间约束是否能降低丢包和重路由。

4. **节点或链路故障场景**  
   验证算法在异常情况下的鲁棒性。

这些场景最好不要只写名字，还要给出可执行的设置。我现在初步想这样设：

| 场景 | 具体设置 | 主要想验证什么 |
|---|---|---|
| 低负载 | 包到达率低，队列基本不积压 | 验证方法不会因为绕路导致明显变差 |
| 中等负载 | 随机源目的对，链路利用率中等 | 验证综合性能 |
| 热点高负载 | 多个源同时向少数地面站/区域发送业务 | 验证队列感知和负载均衡 |
| 链路频繁断开 | 缩短部分 ISL 可用窗口或模拟跨轨链路断开 | 验证链路寿命约束 |
| 故障场景 | 随机提高部分链路中断概率或直接关闭链路 | 验证可靠性和恢复能力 |

这里我最关键的是高负载热点场景和链路频繁断开场景，因为这两个场景最能体现本文和普通最短路/普通 MAPPO 的差别。

为了避免场景描述太虚，我先给一个可执行的初始设置：

| 参数 | 初始设想 | 作用 |
|---|---|---|
| 星座规模 | 先从 100-500 颗卫星仿真，再扩展 | 先保证实验能跑，再看规模泛化 |
| 每颗卫星邻居数 | 约 4 条 ISL | 对应常见前后/左右星间链路结构 |
| 低负载 | 平均链路利用率 `<30%` | 队列不明显，主要看是否绕路 |
| 中等负载 | 平均链路利用率 `30%-60%` | 观察综合性能 |
| 高负载 | 平均链路利用率 `>70%` | 队列和拥塞开始明显 |
| 热点流量 | 例如 60%-80% 流量指向少数目的区域/地面站 | 模拟热点服务区 |
| 快断链路 | 设置部分链路 `T_rem < T_safe` 或频繁变化 | 验证链路寿命过滤 |
| 故障链路 | 随机选 5%-10% 链路提高中断概率 | 验证可靠性 |

这些数值不是最终定死的，而是我为了让实验可执行先设的范围。后面可以根据仿真规模和链路容量调整。

### 6.3 评价指标

主要指标包括：

- 平均端到端时延；
- 吞吐量；
- 投递成功率；
- 丢包率；
- 平均队列长度；
- 链路负载均衡度；
- 路由切换次数；
- 控制消息开销。

### 6.4 消融实验

为了证明每个模块不是简单堆上去的，可以做消融实验：

- 去掉队列感知；
- 去掉链路负载项；
- 去掉链路剩余可用时间；
- 去掉动作掩码；
- 去掉全局协同奖励。

通过这些实验观察端到端时延、丢包率、队列长度和路由稳定性是否变差，从而说明各模块的作用。

消融实验我想具体看这些变化：

| 去掉的模块 | 预期变差的指标 | 说明 |
|---|---|---|
| 去掉队列感知 | 平均队列长度、最大队列长度、端到端时延 | 说明队列项是否真的缓解拥塞 |
| 去掉链路负载 | 链路负载方差、最大链路利用率 | 说明负载项是否真的让流量更均衡 |
| 去掉链路剩余可用时间 | 重路由次数、丢包率、链路断开导致的失败次数 | 说明链路寿命约束是否有效 |
| 去掉动作掩码 | 无效动作比例、训练收敛速度、丢包率 | 说明过滤机制是否减少无效探索 |
| 去掉全局协同奖励 | 全网时延、负载均衡度、吞吐量 | 说明全局目标是否改善多智能体协作 |

这样老师如果问“这个因素到底影响多大”，我后面就可以通过这些消融图回答，而不是只说“理论上有帮助”。

## 8. 研究思维画布九步法梳理

为了让这个方向更清楚，我按照“研究思维画布九步法”重新梳理了一下。

### 8.1 用户场景

应用于高动态低轨卫星网络中的星间路由场景。研究对象是由大量 LEO 卫星组成的时变网络，每颗卫星作为一个路由决策节点，需要在只掌握自身和一跳邻居状态的情况下完成数据包转发。

这个场景下比较典型的问题是：卫星移动快、链路变化快、业务流量不均匀，而且地面中心很难实时控制所有卫星的路由决策。

### 8.2 研究需求

目前的痛点主要是：在低控制开销和低计算开销的前提下，很难让每颗卫星在线计算出既低时延、又不拥塞、还不容易断开的下一跳。

这里“控制开销”我想具体理解成三类：

1. **状态收集开销**：全网链路状态、队列状态、链路负载需要被收集；
2. **状态分发开销**：如果每颗卫星都要知道全网状态，就要把这些信息传播到大量节点；
3. **重计算开销**：拓扑或流量变化后，需要重新计算路由。

按前面的粗略估算，万颗级星座一次全网链路状态快照就可能是数百 KB 级，如果分发给所有卫星，总量就是 GB 级；如果几秒一次更新，控制流量就会非常可观。这也是我想做分布式局部决策的原因。

具体来说：

- 如果依赖全局链路状态，控制开销比较大；
- 如果只用最短路径，容易忽略队列积压和链路负载；
- 如果不考虑链路剩余可用时间，可能选到即将断开的链路；
- 如果各卫星只追求局部最优，可能导致全局拥塞或路由震荡。

### 8.3 核心卖点

我目前想突出的亮点有三个：

1. **基于队列和负载状态进行下一跳决策**  
   不只看传播时延，也看下一跳是否拥塞。

2. **基于链路寿命约束进行动作过滤**  
   在策略选择前先过滤掉即将断开、可靠性低或不可用的链路。

3. **基于 PS-MAPPO 实现集中训练、分布式执行**  
   训练时利用全局状态学习协同策略，执行时每颗卫星只用局部观测独立决策。

### 8.4 解决方案

整体解决思路是：

先把 LEO 网络建模为时变图，再把每颗卫星建模为智能体。每颗卫星通过局部观测和邻居 Hello 包获取状态信息，包括自身队列、邻居队列、链路负载、剩余带宽、可靠性和链路剩余可用时间。

在真正选择下一跳之前，先用动作掩码过滤不可用或高风险链路。然后使用 PS-MAPPO 的共享 Actor 在候选邻居中选择下一跳。训练阶段使用集中式 Critic 和全局奖励，引导多个卫星智能体形成协同路由策略。

### 8.5 重要意义

如果这个方案有效，意义主要有三点：

- 在不依赖地面中心实时全局控制的情况下，实现星上分布式路由；
- 在高负载场景下缓解热点拥塞，降低队列积压和端到端时延；
- 在链路频繁变化场景下提前规避快断链路，减少丢包、重路由和路由震荡。

这里的“有效”我不能只凭感觉说，至少要对应到下面这些结果：

| 意义 | 需要看到的实验现象 |
|---|---|
| 分布式路由有效 | 控制开销明显低于全局链路状态类方法，单次决策时间较低 |
| 缓解拥塞 | 最大队列长度下降，链路负载方差下降，95% 分位时延下降 |
| 提升稳定性 | 链路断开导致的丢包下降，重路由次数下降，路由切换率不过高 |
| 保持吞吐 | 高负载下投递成功率和吞吐量不低于对比算法 |

### 8.6 代价

这个方案也不是没有代价，主要包括：

- 需要先在仿真平台进行集中训练；
- 仿真环境和真实卫星网络之间可能存在差异；
- 多智能体训练时间可能较长；
- Hello 包邻居状态交换会带来一定控制开销；
- 奖励函数权重和动作过滤阈值需要调参。

这些代价也要量化，不然老师可能会问“你省下来的开销会不会被训练和 Hello 包抵消”。我现在打算这样看：

| 代价 | 怎么量化 |
|---|---|
| 训练时间长 | 记录训练收敛轮数、训练总时长 |
| 仿真到真实有偏差 | 在不同星座规模、不同流量模式下测试泛化 |
| Hello 控制开销 | `control_bytes / data_bytes` |
| 星上计算开销 | 单次 Actor 推理时间、模型参数量 |
| 调参成本 | 做权重敏感性实验，观察性能是否稳定 |

如果最后发现 Hello 控制开销超过 `5%`，或者 Actor 推理时间太长，那这个方案的“低开销”说法就站不住，需要调整 Hello 周期或压缩模型。

### 8.7 核心技术指标

后续实验可以从以下指标验证：

1. **时延性能**：平均端到端时延、95% 分位时延；
2. **吞吐和可靠性**：吞吐量、投递成功率、丢包率；
3. **拥塞情况**：平均队列长度、最大队列长度、链路负载均衡度；
4. **稳定性**：路由切换次数、重路由次数、链路断开后的恢复时间；
5. **开销**：Hello 包控制开销、单次路由决策时间、模型参数量。

其中我觉得最关键的不是单独看平均时延，而是要同时看：

```text
平均端到端时延 + 最大队列长度 + 丢包率 + 路由切换次数
```

因为如果只看平均时延，可能看不出拥塞和链路失效问题；如果只看吞吐，也可能忽略局部热点队列。

我后面判断“好不好”不能只靠主观描述，可以先设一组实验判断标准：

| 指标 | 比较好的状态 | 说明 |
|---|---|---|
| 平均端到端时延 | 相比 Dijkstra/Q-routing 明显下降，或者不高于最短路太多 | 证明没有因为绕路导致整体时延恶化 |
| 95% 分位时延 | 明显下降 | 说明尾部拥塞减少，不只是平均值好看 |
| 投递成功率 | `>95%` 比较理想 | 说明大部分包能送达 |
| 丢包率 | `<1%` 较好，`>5%` 较差 | 丢包高说明拥塞或链路失效严重 |
| 平均队列长度 | 低于队列上限的 `30%` 较健康 | 说明整体拥塞不严重 |
| 最大队列长度 | 不长期超过队列上限的 `70%` | 说明没有持续热点节点 |
| 链路负载变异系数 | `<0.3` 较均衡，`>0.6` 热点明显 | 说明流量是否集中 |
| 路由切换率 | `<0.1` 稳定，`>0.3` 可能震荡 | 说明路径是否频繁变化 |
| 控制开销比例 | `<1%` 很低，`1%-5%` 可接受 | 证明不是靠大量控制消息换性能 |

这些阈值不是最终结论，而是实验前的判断标准。真正写论文时，需要结合仿真参数和对比算法结果再调整。

### 8.8 进一步改进空间

目前这个方案后续还可以继续改进：

- 泛化能力还需要验证，比如星座规模、流量模式变化较大时是否仍然有效；
- 链路剩余可用时间的估计方法还需要进一步明确；
- 奖励函数权重可能比较敏感，需要做参数分析；
- 如果真实星上计算资源有限，后续可能需要做模型压缩；
- 当前主要考虑单一业务，后续可以扩展到多业务 QoS 路由。

这里面我觉得最重要的不是“以后还能做什么”，而是哪些点可能真的影响论文结果：

| 不完善点 | 可能造成的影响 | 后续怎么补 |
|---|---|---|
| 泛化不足 | 训练好的策略换星座规模后性能下降 | 做不同规模、不同负载的泛化实验 |
| 链路寿命估计不准 | 误判快断链路，导致不必要绕路或丢包 | 用轨道预测/时间片拓扑计算 `T_rem` |
| 奖励权重敏感 | 某组参数下好，换负载后变差 | 做权重敏感性和消融实验 |
| 星上算力有限 | Actor 推理慢或模型太大 | 参数共享、模型压缩、限制输入维度 |
| 单业务假设 | 不同业务 QoS 需求无法区分 | 后续加入业务优先级或多目标奖励 |

### 8.9 回到原点

这个方向短期内更适合先在仿真平台中验证，比如自建 LEO 网络仿真环境，或者结合 ns-3、STK 等工具生成拓扑和链路变化。等仿真结果稳定后，再考虑原型平台或更真实的系统验证。

所以我现在的阶段目标不是直接部署到真实卫星，而是先证明这个建模和路由机制在仿真环境中确实能改善时延、拥塞和链路失效问题。

## 9. 目前总结

目前我的整体思路是：

> 先把 LEO 卫星网络建模为时变图，再把每颗卫星建模为一个智能体。每颗卫星只利用自身状态、一跳邻居状态和相邻链路状态进行下一跳选择。为了避免拥塞和链路失效，在状态和奖励中加入队列长度、链路负载、链路可靠性和链路剩余可用时间，并通过动作掩码过滤明显不可用或高风险链路。方法上采用 PS-MAPPO，实现训练阶段全局协同、执行阶段分布式自治。实验上通过对比实验和消融实验验证该方法在时延、丢包率、负载均衡和路由稳定性方面的效果。

现在这条思路里，我觉得最关键的逻辑链是：

```text
LEO 高动态 + 万颗级规模
        ↓
全局状态维护和集中式最短路开销大
        ↓
只能依赖局部观测和邻居交换
        ↓
局部最短路又容易忽略队列、负载、链路寿命
        ↓
加入队列感知、链路寿命约束和动作掩码
        ↓
用 PS-MAPPO 在训练阶段学全局协同，执行阶段保持分布式
        ↓
通过时延、队列、丢包、重路由、控制开销验证是否有效
```

后续需要进一步细化的地方包括：

- 状态空间具体如何归一化和组织；
- 链路剩余可用时间如何计算；
- 奖励函数各项权重如何设置；
- 仿真星座和业务流量如何生成；
- 与已有 LEO 路由算法的公平对比方式。

我下一步最应该做的不是继续加概念，而是把三个东西定下来：

1. **仿真场景参数**：星座规模、链路容量、业务到达率、热点流量比例；
2. **奖励函数参数**：`D_target`、`Q_max`、`T_safe` 和各项权重；
3. **消融实验表**：每个模块去掉后观察哪些指标变化。

这样老师再问“到底怎么证明”，我就能直接指到具体实验和指标，而不是只说这个因素有帮助。
