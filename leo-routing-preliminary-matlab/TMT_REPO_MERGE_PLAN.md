当前已经定位到了三个算法仓库：

- `F:\cleanmarl`
- `F:\on-policy`
- `F:\BenchMARL`

但你提到的主参考仓库：

## 2. 找到 tmt 仓库以后，优先看什么

如果后续定位到 `tmt-123-bit/leo-routing`，建议优先看四类内容。

### 第一类：README / 研究说明

重点看：

- 它怎么描述问题建模
- 有没有明确写 RTMDP / Dec-POMDP / MAPPO
- 它的状态、动作、reward 是怎么组织的
- 它的实验场景和指标有哪些

### 第二类：环境代码

重点看：

- 有没有单独的环境层文件
- 它的 observation / action mask 是怎么做的
- 有没有局部状态和全局状态区分
- 是否已经有真实拓扑输入逻辑

### 第三类：训练脚本

重点看：

- 是直接自写 MAPPO，还是基于成熟库改的
- 输入输出格式和当前 `leo_marl_env.py` 是否接近
- 是否已经实现 policy 保存和统一评测

### 第四类：实验输出与配置

重点看：

- 参数文件或 config
- 场景定义
- 输出 CSV / 图表
- 消融实验表格

## 3. 找到以后最推荐的并入方式

### 不推荐的方式

不建议：

```text
把当前项目全部删掉，用 tmt 仓库完全替代
```

原因：

- 当前项目已经有完整的 MATLAB 前期验证、Python 环境、评测入口、轻量学习闭环、cleanmarl bridge；
- 如果直接替换，容易把已经梳理好的论文映射打乱。

### 推荐的方式

更推荐：

```text
把 tmt 仓库当成“设计和实现对照源”，逐项并入
```

也就是：

1. 保留当前项目目录和评测入口不变；
2. 从 tmt 仓库里抽取更优的：
   - 状态字段
   - reward 设计
   - action mask 细节
   - 配置参数
   - 实验组织方式
3. 逐步并到：
   - `leo_marl_env.py`
   - `run_python_experiments.py`
   - README / 说明文档

这样做的好处是：

- 当前项目不会失控；
- 论文主线保持一致；
- 更容易说清楚“哪些部分来自主参考仓库启发，哪些是自己整理后的工程实现”。

## 4. tmt 仓库一旦找到，建议立刻做的三件事

### Step 1
先只做结构审阅，不急着改代码。

先确认：

- 它到底是环境仓库、训练仓库，还是研究笔记仓库；
- 它是不是比当前 `leo_marl_env.py` 更成熟；
- 它是不是已经接了某个具体 MAPPO 实现。

### Step 2
把有价值的信息写进一个“对齐表”。

例如：

| tmt 仓库模块 | 当前项目模块 | 处理方式 |
|---|---|---|
| 状态定义 | `leo_marl_env.py` | 合并 / 保留 / 替换 |
| reward 设计 | `_local_reward()` | 调整 |
| 场景参数 | `SCENARIOS` | 扩充 |
| 训练入口 | `train_*` | 参考或替换 |

### Step 3
优先动环境和配置，不优先动训练仓库。

先把：

- 观察空间
- 动作掩码
- reward
- 真实拓扑输入
- 场景配置

并进当前项目。

训练脚本是否替换，应该放后面判断。
