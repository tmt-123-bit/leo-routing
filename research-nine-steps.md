# 研究思维画布九步法

题目建议：

**基于 PS-MAPPO 的队列感知与链路寿命约束低轨卫星网络分布式动态路由研究**

## 1. 用户场景

应用于大规模低轨卫星网络中的星间链路动态路由场景，尤其适用于卫星高速运动、拓扑频繁变化、业务流量不均衡、局部链路容易拥塞的场景。

研究对象是由多颗 LEO 卫星组成的时变网络。每颗卫星作为一个智能体，只能获取自身状态、一跳邻居状态以及相邻链路状态，在不依赖地面中心实时全局控制的情况下，自主选择数据包的下一跳转发节点。

典型场景包括：

- 星间链路频繁建立和断开。
- 热点区域流量突发，部分卫星队列积压。
- 地面站接入流量不均衡。
- 卫星只能进行有限邻居信息交换。
- 需要快速完成路由决策，不能频繁运行全局最短路算法。

## 2. 研究需求

当前 LEO 卫星网络动态路由主要存在三个痛点。

第一，拓扑变化快，集中式路由开销高。LEO 卫星高速运动，星间链路随时间变化。如果依赖地面中心收集全网链路状态并计算路由，会带来较大的控制开销和较慢的收敛速度。

第二，传统路由缺乏拥塞感知能力。最短路径或最小跳数路由通常只考虑传播时延或链路距离，但在高负载情况下，真正影响端到端时延的是队列积压和链路拥塞。

第三，路由决策没有充分考虑链路剩余可用时间。某些链路当前可用，但可能很快断开。如果只根据当前链路质量选择下一跳，可能导致数据包转发途中链路失效，引起重路由、丢包或额外时延。

本文需要解决的问题是：

**在仅依赖局部观测和邻居信息交换的条件下，如何让每颗卫星快速、分布式地选择下一跳，并同时规避拥塞链路和即将断开的链路。**

## 3. 核心卖点

第一，提出队列感知的分布式路由决策机制。在下一跳选择中不仅考虑传播时延和链路带宽，还显式引入下一跳卫星队列长度、链路负载率等拥塞状态。

第二，提出链路剩余可用时间约束的动作过滤机制。在智能体选择下一跳之前，先过滤掉即将断开、可靠性较低、带宽不足或队列过长的邻居链路。

第三，采用基于 PS-MAPPO 的 CTDE 模型。训练阶段利用全局网络状态和全局协同奖励函数，使所有卫星智能体学习符合全网性能目标的路由策略；执行阶段每颗卫星只依赖本地和邻居状态独立决策。

## 4. 解决方案

将 LEO 卫星网络建模为时变图：

```text
G(t) = (V, E(t), X(t))
```

其中，`V` 表示卫星节点集合，`E(t)` 表示时刻 `t` 的可用星间链路集合，`X(t)` 表示节点和链路状态。

每颗卫星作为一个智能体。卫星 `i` 在转发数据包时，获取局部观测：

```text
o_i(t) = {q_i, q_j, R_ij, d_ij, rho_ij, P_out_ij, T_rem_ij, dest}
```

其中：

- `q_i`：当前卫星队列长度。
- `q_j`：邻居卫星队列长度。
- `R_ij`：链路剩余带宽。
- `d_ij`：链路传播距离或传播时延。
- `rho_ij`：链路负载率。
- `P_out_ij`：链路中断概率。
- `T_rem_ij`：链路剩余可用时间。
- `dest`：目的节点或目的区域编码。

动作过滤：

```text
A_i(t) = {j in N_i(t) |
          R_ij > R_min,
          q_j < Q_max,
          P_out_ij < P_max,
          T_rem_ij > T_safe}
```

PS-MAPPO Actor 在候选动作集合中选择下一跳：

```text
a_i(t) = pi_theta(o_i(t), mask_i(t))
```

训练阶段使用集中式 Critic：

```text
V_phi(s(t))
```

其中 `s(t)` 是全局网络状态，只在训练阶段使用，执行阶段不部署到星上。

## 5. 重要意义

第一，降低大规模 LEO 网络的路由控制开销。相比集中式路由，本文方法不需要频繁向地面中心上报全网链路状态，也不需要在每次拓扑变化后重新计算全局路径。

第二，提升高负载场景下的抗拥塞能力。通过引入下一跳队列长度和链路负载率，路由策略可以主动避开拥塞节点和热点链路。

第三，增强动态拓扑下的路径稳定性。通过链路剩余可用时间过滤机制，卫星可以提前规避即将断开的链路。

总体意义：

**实现一种局部决策、邻居协作、全局目标一致的 LEO 卫星网络动态路由机制。**

## 6. 代价

第一，训练成本较高。PS-MAPPO 需要在仿真环境中进行大量训练，需要构建 LEO 星座拓扑、业务流量模型、链路变化模型和队列模型。

第二，仿真到真实环境存在差距。训练环境中的链路变化、流量分布和故障模型可能无法完全等同于真实卫星网络。

第三，邻居状态交换会产生额外控制开销。每颗卫星需要通过 Hello 包周期性交换队列长度、链路负载和链路可用时间等信息。

第四，奖励函数权重需要调节。端到端时延、吞吐量、负载均衡、丢包率和路由稳定性之间可能存在冲突。

## 7. 核心技术指标

时延性能：

- 平均端到端时延。
- 95% 分位端到端时延。
- 平均单跳排队时延。

吞吐与可靠性：

- 网络总吞吐量。
- 投递成功率。
- 丢包率。

拥塞与负载均衡：

- 平均队列长度。
- 最大队列长度。
- 链路负载方差。
- 节点负载方差。

动态稳定性：

- 路由切换次数。
- 链路断开后的恢复时间。
- 路由震荡次数。
- 拓扑变化后的收敛时间。

控制与计算开销：

- Hello 包控制开销。
- 单次路由决策时间。
- 星上模型参数量。
- 与 Dijkstra、Q-routing、普通 MAPPO 的复杂度对比。

## 8. 进一步改进空间

- 提高模型在不同星座规模、轨道参数和流量分布下的泛化能力。
- 压缩策略网络，使其更适合星上有限计算资源。
- 引入在线微调机制，但需要避免在线学习导致路由不稳定。
- 扩展到多业务 QoS 路由。
- 考虑链路干扰、节点异常和攻击场景下的鲁棒性。

## 9. 回到原点

该方向可以直接服务于真实 LEO 卫星网络，但短期内更适合作为仿真验证型研究。

实验上应验证：

- 是否比 Dijkstra、虚拟拓扑法、Q-routing 更低时延。
- 是否能缓解热点拥塞。
- 是否能减少链路断开带来的丢包和重路由。
- 是否能在只依赖局部信息的情况下接近全局优化效果。

一句话总结：

**本文解决的是 LEO 网络中局部信息下的拥塞感知和链路失效规避问题，方法是队列感知 + 链路寿命约束 + PS-MAPPO-CTDE。**
